# Приватность и хранение данных (retention)

> Last reviewed: 2026-07-18

Что приложение хранит, где, как долго и как удаляет — с упором на **финансовые ПДн** (банковские
выписки: счета, суммы, контрагенты, назначения). Слайс #245; смежное — приватный feedback-репо #284.

## Принцип: выписки у себя не храним

Источник правды по операциям/сделкам/инвойсам — **Bitrix24-портал клиента**. В нашей БД **нет
таблицы выписок/операций**: разобранные операции пишутся **делами в CRM портала** (`crm.activity.configurable.add`),
дедуп — маркером в самом B24 (#259). Содержимое выписки проходит через нас **транзитно** (очереди),
не оседает в долгом хранилище.

## Что реально лежит в нашей Postgres

| Таблица | Что | ПДн? | Чистка |
|---|---|---|---|
| `portal_tokens` | OAuth-токены портала (refresh — AES-256-GCM) | нет выписок | удаление на `ONAPPUNINSTALL` |
| `portal_tombstone` | метка удаления портала (порядок событий #77) | нет | остаётся (анти-воскрешение) |
| `bank_tokens` | банк-OAuth токены счёта (refresh шифрован) | нет выписок | удаление на `ONAPPUNINSTALL` |
| `allocation_fact` | факт «платёж → сущность» (`fact_key` = наш счёт + docId + id/тип цели; сумма/контрагент — нет) | наш счёт + docId операции (не сама выписка) | удаление на `ONAPPUNINSTALL` |
| `import_result` | сводка **последнего** прогона (счётчики) | агрегаты, не операции | удаление на `ONAPPUNINSTALL` |
| `metrics_counter` | пожизненные счётчики портала | агрегаты | удаление на `ONAPPUNINSTALL` |

**Удаление приложения (`ONAPPUNINSTALL`) стирает ВСЁ** по порталу (`deletePortal`): токены, факты,
статус, метрики, банк-токены. Флаг «очистить данные» не смотрим — чистим всегда.

## Транзитные ПДн: payload'ы очередей в Redis

Само содержимое выписки живёт **недолго в Redis** как payload джоба:

- `file-parse` — загруженный **файл** выписки (base64, ≤ ~2.7 МБ);
- `crm-sync` — нормализованный `StatementItem[]` (контрагент / счёт / сумма / назначение).

Оба enqueue'ятся с **ограниченным по возрасту** удержанием (`STATEMENT_JOB_RETENTION`,
`server/queue/producers.ts`, #245): выполненные — **1 час** (кап 50), упавшие — **сутки** (кап 200, для
отладки). То есть финансовые ПДн **не копятся** под count-дефолтом (1000/5000 payload'ов), а вытесняются
по возрасту **или** капу — что раньше. ⚠ Удаление у BullMQ **ленивое** (не по таймеру): чистка состояния
срабатывает, когда в этой очереди **завершается следующий** джоб. Поэтому на **простаивающем** портале
«последние» ≤50 completed / ≤200 failed payload'ов могут полежать дольше 1 ч/суток — до следующего
завершения в очереди; жёсткая верхняя граница — **count-кап** (50/200), а не абсолютное время.
Опциональный follow-up для гарантии по стенным часам — периодический `queue.clean` на крон-инстансе.

`bank-fetch` содержимого выписки не несёт (payload = «наш счёт + окно дат»; наш счёт, не контрагент) —
остаётся на count-дефолте. **`b24-events`** несёт в install-payload **OAuth access-токен портала в
открытом виде** (+ refresh шифрован): раз событие применено (токен сохранён), удерживать креды незачем,
поэтому completed-джоб удаляется **сразу** (`CREDENTIAL_JOB_RETENTION`, `removeOnComplete: true`), failed —
ограниченно (сутки/200) для отладки установок. Access-токен короткоживущий (~1 ч), refresh — шифрован.

Redis — на изолированной сети `queuenet` (`internal: true`), наружу не смотрит.

## Логи

Внешние поля (счёт/назначение/имя контрагента) при логировании прогоняются через `logSafe`
(control-char-стрип) — от инъекции строк лога. В диагностических логах операций
(`[recognize]`/`[resolve]`/`[allocate]`) фигурируют **счёт + docId + распознанные идентификаторы**
(извлечённые из назначения платежа — номера заказов/оплат/счетов); **суммы в логи не пишутся**.
Хранение/ротацию логов контейнеров задаёт хостовый docker (`json-file`) — настроить ограничение
размера/срока на сервере (runbook — `OPERATIONS.md`).

## Открытые пункты (follow-up)

- **TTL «спящих» агрегатов** (#245): `import_result`/`metrics_counter`/`allocation_fact` живут до
  uninstall — для установленного, но давно неактивного портала можно ввести необязательный TTL/архивацию
  (сейчас не критично: без выписок, только id-ссылки/счётчики).
- **Приватный feedback-репо** (#284): файл выписки, приложенный к отзыву для воспроизведения, —
  срок хранения/подчистка в приватном репо-приёмнике (по умолчанию PII в публичный issue **не** кладём —
  см. `FEEDBACK.md`).
- **Маскировка идентификаторов в логах**: суммы в логи не идут, но счёт + распознанные номера
  (заказ/оплата/счёт из назначения) — да; рассмотреть маскирование при разборе инцидентов на чужой стороне.
- **Отладка упавших импортов**: payload упавшего statement-джоба ограничен **и** возрастом (сутки),
  **и** капом (200) — что раньше; на объёмном портале старый упавший payload может уже уйти. Криминалистика
  по последнему прогону живёт в Postgres (`import_result.errors` + метрики), не в Redis — там и смотреть.
- **Гарантия по стенным часам**: при необходимости жёсткого TTL (не ленивого) — периодический `queue.clean`
  для statement-очередей на крон-инстансе.
