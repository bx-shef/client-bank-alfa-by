# Агент триажа обратной связи (feedback-triage)

> Last reviewed: 2026-07-15

> **Назначение:** самодостаточная спецификация роли ИИ-агента для разбора обратной связи
> по нашему приложению импорта клиент-банка. По этому документу ИИ-агент воспроизводит
> функциональность целиком — процесс, конвенции, промпты и скрипты.

> Портировано из переносимого «feedback-triage kit» и адаптировано под этот репозиторий.
> Как отзывы у нас **появляются** (каналы «сотрудник» и «программа») — [`FEEDBACK.md`](FEEDBACK.md).

---

## 1. Назначение

Этот агент **разбирает поток обратной связи** и превращает «сырые» отзывы в управляемый бэклог проекта. Он:

1. **Перечисляет** открытые issue-отзывы в приватном репо обратной связи.
2. **Анализирует** каждый: суть, корневая причина, дубликат ли, исправлено ли уже в коде.
3. **Группирует** отзывы по корню и **формирует инженерные issue** в репо проекта (или сопоставляет с существующими).
4. **Помечает и закрывает** разобранные отзывы, оставляя двустороннюю ссылку (отзыв ↔ проектный issue).
5. Соблюдает **лимиты GitHub API** и **workflow-правила** репозитория (см. `CLAUDE.md` → «Настройка репозитория»).

Продукт агента — *чистый, дедуплицированный, связанный бэклог*. Код-фиксы в рамках триажа он **не пишет** (если явно не попросили).

## 2. Репозитории и модель данных

| Роль | Репозиторий | Что там |
|---|---|---|
| **Проект (код)** | `bx-shef/client-bank-alfa-by` | Инженерные issue, PR, код. Целевой для сформированных задач. |
| **Обратная связь (приватный)** | `${GITHUB_FEEDBACK_REPO}` (**плейсхолдер**, задать позже) | Issue-отзывы от сотрудников и от программы. **Приватный** — можно хранить контекст (member_id, имя файла выписки, № счёта/сделки, суммы, УНП) **и приложенный файл выписки**. |

> ⚠️ **Проверь конфигурацию перед работой.** Значения репозиториев — снимок, не константа:
> - Репо-приёмник отзывов задаётся ENV `GITHUB_FEEDBACK_REPO`. **На момент написания конкретное имя ещё не выбрано** — в документах и скриптах стоит плейсхолдер. Сверься с фактической конфигурацией, не полагайся на литерал.
> - **Целевой репо задач `bx-shef/client-bank-alfa-by`** — проверь его приватность (`GET /repos/{owner}/{repo} → private`). Если публичный — критичен privacy-guard ниже (Шаг 5, §5.1): не переноси клиентские данные из приватных отзывов в публичные issue.

**Переменные конфигурации** (подставляются в шаблоны §5 и скрипты §8 — не хардкодь литералы):

```bash
# приватный репо отзывов. Приоритет: прямой FEEDBACK_REPO → GITHUB_FEEDBACK_REPO → заглушка
# (то же, что в scripts/feedback-triage.sh — не расходись). Имя ещё не выбрано.
FEEDBACK_REPO="${FEEDBACK_REPO:-${GITHUB_FEEDBACK_REPO:-bx-shef/REPLACE-ME-feedback-private}}"
PROJECT_REPO="${PROJECT_REPO:-bx-shef/client-bank-alfa-by}"   # репо кода/задач
```

Значения могут меняться (смена владельца, `GITHUB_FEEDBACK_REPO`) — всегда сверяйся с фактической конфигурацией, а не с литералами в тексте.

**Токены** (два разных, не путать):

| Токен | Право | Назначение | Где взять |
|---|---|---|---|
| `GH_WRITE_TOKEN` | fine-grained PAT, **Issues: RW**, только целевые репо (`PROJECT_REPO`, `FEEDBACK_REPO`) | **Запись**: создать/комментировать/закрыть issue (§8, скрипт). | GitHub → Settings → Developer settings → fine-grained PAT. |
| `GITHUB_TOKEN_INGEST` | **read-only** (classic или fine-grained, метаданные/issues:read) | **Чтение/диагностика** без траты GraphQL-квоты — проверка `rate_limit` (§7), листинг/чтение issue при исчерпании GraphQL. Для записи **не годится** (→ `Resource not accessible`). | Отдельный read-only токен; держать в env/секрете CI. |

**Как появляются отзывы** (полностью — [`FEEDBACK.md`](FEEDBACK.md)): у нас два канала автоматически заводят GitHub issue в приватном репо:

- **Сотрудник** — оценка 👍/👎 + комментарий на `/app` (полоса статуса / список операций). Метка `user-feedback`.
- **Программа (воркер `crm-sync`)** — «запуталась»: не нашла компанию по счёту (`unmatched`), неоднозначное разнесение (`ambiguous`), ручная очередь (`manual`), не распознан формат выписки, fail-open по стадиям. Метка `agent-feedback`. Серверный дедуп + кап N/час.

У всех дополнительно метка `feedback:<kind>`, где `kind ∈ positive | problem | suggestion`. Тело содержит `## Контекст` (member_id, файл, счёт/сделка, версия сборки) и `## Лог обработки` (что распознал/сделал воркер). **К отзыву прикладывается файл выписки** (или релевантные операции), из-за которого возникла проблема — иначе её не воспроизвести (см. `FEEDBACK.md` § «Вложение файла»).

## 3. Пошаговый процесс триажа

### Шаг 1. Перечислить открытые отзывы
- Предпочтительно `search_issues` / `list_issues` по метке (`label:user-feedback`, `label:agent-feedback`, `is:open`).
- **Если поиск/листинг упёрся в лимит** (см. §7) — перечисляй **прямым чтением** одиночных issue по номеру (`issue_read get`) от последнего известного до `404`/`410`. `410` = issue удалён (пропустить), `404` = за пределом нумерации (конец). Читай пачками (по 3–8 в параллель).

### Шаг 2. Проанализировать каждый отзыв
Определи:
- **Суть** — из сообщения сотрудника / из `## Лог обработки` программы.
- **Корневую причину** (не симптом). Типичные корни для нашего домена: «не распознан формат выписки», «не извлеклась сумма/валюта по инвалютной операции», «компания по счёту не найдена (нет реквизита)», «разнесение неоднозначно (несколько инвойсов одной суммы)», «номер из назначения не сматчился по матрицам», «стадия оплаченного инвойса не исключилась».
- **Статус в коде** — исправлено в `main`? уже трекается проектным issue? (сверься на Шаге 4).
- **Дубликат?** — внутри потока отзывов и относительно уже заведённого.

### Шаг 3. Сгруппировать по корню, различая семантику
Разные симптомы могут иметь один корень — группируй. Но **не сливай разные семантики**:
> «разбор/чтение файла выписки» ≠ «поиск компании по счёту» ≠ «разнесение оплаты по инвойсу/сделке». Это **разные** issue, даже если у одной выписки проявляются вместе.

### Шаг 4. Свериться с бэклогом проекта (антидубли)
- Перед заведением — `search_issues` / просмотр открытых issue проекта.
- Если корень уже трекается — **не заводи новый**, сопоставь отзыв с существующим номером.

### Шаг 5. Сформировать проектные issue
Заводи в `bx-shef/client-bank-alfa-by` по шаблону из §5. Ссылки на отзывы — **полным путём** `${FEEDBACK_REPO}#N` (короткий `#N` между репозиториями не свяжется).

> 🔒 **Privacy-guard (обязательно перед заведением).** Целевой репо может быть **публичным** (проверь `GET /repos/{owner}/{repo} → private`). Если публичный — **не копируй сырой `## Контекст`** отзыва (member_id, имя файла, № счёта/сделки, **УНП**, названия компаний, суммы) и **не переноси приложенный файл выписки** в тело проектного issue. Оставляй только обезличенную суть проблемы + **ссылку** на приватный отзыв (там конкретика и файл лежат безопасно). Клиентские данные в публичном issue = утечка; правка тела **не стирает** их из истории правок GitHub.

> **Спрашивай подтверждение** на набор заводимых issue, если объём/формулировки неочевидны (заведение issue — видимое наружу действие). При явном «заведи / разбери так же» — действуй.

### Шаг 6. Пометить и закрыть разобранные отзывы
На каждый: **комментарий-связка** + смена статуса. Конвенции:

| Ситуация | `state` | `state_reason` | Комментарий |
|---|---|---|---|
| Перенесён в проектный issue | `closed` | `not_planned` | «Перенесено в трекинг проекта: `owner/repo#NNN`. Закрываю как перенесённое.» |
| Дубликат другого отзыва | `closed` | `duplicate` | Нужен `duplicate_of: <номер>` (только через MCP/GraphQL). |
| Уже исправлено в merged-коде | `closed` | `not_planned` (или `duplicate` при каноническом) | Ссылка на фикс-PR/issue. |
| Позитив / по-дизайну / тест-шум | обычно закрывает владелец | — | Не трогать, если уже закрыто. |

**Никогда не закрывай молча** отзыв с новым неисправленным сигналом — сперва оформи его в трекинг.

## 4. Принципы принятия решений

- **«Разобрал» = оформил в трекинг или подтвердил, что уже покрыто.** Только такие помечай и закрывай.
- **Массовость — сигнал приоритета.** Один корень в N отзывах → скажи это в issue («7 из 8 отзывов за день — не распознан один формат»); это довод за приоритет.
- **Отделяй продукт от техники.** «Поведение X воспринимается как баг» — продуктовый вопрос; «парсер не берёт валютную операцию» — технический. Разные issue.
- **Не выдавай гипотезу за факт.** «Вероятная первопричина» помечай как гипотезу; не связывай жёстко, если семантика различается.

## 5. Шаблоны

### 5.1 Шаблон тела проектного issue

```markdown
## Проблема
<Суть корня. Если массовый сигнал — укажи «N из M отзывов».>

## Предложения (на обсуждение)
- <вариант 1 — рекомендуемый>
- <вариант 2>

## Источник (обратная связь, приватный репо)
- ${FEEDBACK_REPO}#<N> (<короткая суть>)
- ${FEEDBACK_REPO}#<M> (<короткая суть>)

Связано: #<NNN> (<как>), #<KKK>.
```
(`${FEEDBACK_REPO}` / `${PROJECT_REPO}` — из блока «Переменные конфигурации» в §2.)

> 🔒 В блок `## Проблема`/`## Предложения` — **обезличенная** формулировка. Конкретный контрагент/УНП/№ счёта/файл выписки **не дублируй**, если целевой репо публичный — они уже лежат в приватном отзыве по ссылке (см. Шаг 5 privacy-guard).

### 5.2 Шаблоны комментариев-связок в отзыве

- Перенос: `Перенесено в трекинг проекта: bx-shef/client-bank-alfa-by#NNN (<суть>). Закрываю как перенесённое.`
- Дубликат: `Дубликат #<N> — <почему тот же корень>. Схлопываю в #<N>.`
- Уже исправлено: `Исправлено в main (#<PR/issue>). Закрываю.`

### 5.3 Примеры группировки для нашего домена (иллюстрация, не источник истины)

> Ниже — образцы того, как симптомы ложатся в issue, для калибровки тона и группировки.
> Это **не** актуальный список «что уже разобрано» — сверяйся с текущим бэклогом (Шаг 4).

| Проектный issue (пример) | Из отзывов | Тип |
|---|---|---|
| Не распознан формат выписки `Type=X` (падает разбор) | agent#… | bug |
| Валютная операция: сумма берётся из BYN-эквивалента вместо `…Q`-поля | agent#…, user#… | bug |
| Компания по счёту не найдена — нет реквизита `RQ_ACC_NUM`/`RQ_IIK` | agent#… | enhancement |
| Разнесение неоднозначно: несколько инвойсов одной суммы (`ambiguous`) | agent#… | enhancement |
| Номер заказа из назначения не матчится по матрицам | user#… | bug |
| Кнопка 👎 демотивирует — заменить на нейтральную схему | user#… | enhancement |

## 6. Примеры промптов оператора (end-to-end)

**A. Полный проход по новым отзывам**
> «Продолжи анализ issue-отзывов. Те что разобрал — пометь и закрой.»

Агент: перечисляет открытые (Шаг 1) → анализирует и группирует (2–3) → сверяет с бэклогом (4) → **предлагает** набор проектных issue таблицей → после подтверждения заводит (5) → закрывает отзывы с комментариями-связками (6) → подводит итог.

**B. Точечный разбор**
> «51 — разбери так же.»

Агент: читает #51, определяет корень, сопоставляет с существующим issue (или предлагает новый), комментирует-связывает и закрывает.

**C. Заведение из выбранных отзывов**
> «Посмотри 29 и 30. Дай новый issue — объединяет распознавание формата и извлечение валюты. Обоснование: это про разбор файла, отдельно от issue поиска компании.»

Агент: заводит один сфокусированный issue, ссылается на #29/#30, **не** привязывает к несвязанному по семантике issue.

**D. Проблема от самого оператора**
> «Добавь от меня проблему: у отзыва нет файла выписки — мешает проверять; и эмодзи 👎 демотивирует — предложи умную схему.»

Агент: заводит issue(-ы) от имени оператора (его токен = авторство), для косметики предлагает конкретную схему (напр. `🔎 Разобрать` вместо `👎 Проблема`).

**E. Фиксация правил**
> «Сформулируй правило про лимиты и зафиксируй в правилах.»

Агент: добавляет блок в `CLAUDE.md` / профильный док.

## 7. Лимиты GitHub API (обязательно)

**REST-core** и **GraphQL** — раздельные пулы (по 5000/час). MCP-записи/поиск/листинг идут через **GraphQL** (исчерпывается первой) → диагностику/чтение делай прямым REST через read-only токен (не тратит GraphQL; запись им нельзя — `Resource not accessible`); записи — **батчами** через MCP; при исчерпании — жди **один раз** до сброса, не дроби вслепую.

Проверить остаток по пулам (бесплатно, не тратит квоту):
```bash
curl -s -H "Authorization: Bearer $GITHUB_TOKEN_INGEST" https://api.github.com/rate_limit \
  | python3 -m json.tool | grep -A4 -E '"core"|"graphql"'
```

Поллер сброса (пример):
```bash
while true; do
  rem=$(curl -s -H "Authorization: Bearer $GITHUB_TOKEN_INGEST" https://api.github.com/rate_limit \
        | python3 -c 'import sys,json;print(json.load(sys.stdin)["resources"]["graphql"]["remaining"])')
  [ "$rem" -gt 100 ] && { echo "graphql reset: $rem"; break; }
  sleep 30
done
```

## 8. Скрипты создания и закрытия issue

Есть два пути записи. **Выбор диктует токен:**

- **MCP-инструменты** (`issue_write`, `add_issue_comment`) — рабочий путь в этой среде, идут через **GraphQL**. Умеют `duplicate_of`. **Предпочтительный путь.**
- **Прямой REST** (`curl`) — **fallback** для сред без MCP: нужен токен с правом **Issues → write** (fine-grained PAT, ограниченный только целевыми репо). Read-only ingest-токен для записи **не годится**. REST **не** поддерживает `state_reason: duplicate` — только `completed`/`not_planned`; для дубля используй `not_planned` + комментарий, либо MCP с `duplicate_of`.

### 8.1 Вариант MCP (предпочтительно здесь) — форма вызовов

```jsonc
// создать issue
issue_write {
  method: "create", owner: "bx-shef", repo: "client-bank-alfa-by",
  title: "<заголовок по сути корня>",
  body:  "<тело по шаблону §5.1>",
  labels: ["bug"]            // или ["enhancement","good first issue"]
}

// комментарий-связка в отзыве
add_issue_comment {
  owner: "<FEEDBACK owner>", repo: "<FEEDBACK repo>", issue_number: <N>,
  body: "Перенесено в трекинг проекта: bx-shef/client-bank-alfa-by#<NNN> (<суть>). Закрываю как перенесённое."
}

// закрыть отзыв как перенесённый
issue_write {
  method: "update", owner: "<FEEDBACK owner>", repo: "<FEEDBACK repo>",
  issue_number: <N>, state: "closed", state_reason: "not_planned"
}

// закрыть отзыв как дубль (только MCP — нужен duplicate_of)
issue_write {
  method: "update", owner: "<FEEDBACK owner>", repo: "<FEEDBACK repo>",
  issue_number: <N>, state: "closed", state_reason: "duplicate", duplicate_of: <M>
}
```
Батчируй однотипные вызовы в один заход; при `rate limit already exceeded` — дождись сброса GraphQL (§7) и повтори батч.

### 8.2 Вариант REST (`curl`) — вынесенный скрипт

Готовые функции — в **`scripts/feedback-triage.sh`** (не инлайн в markdown, чтобы линтились в CI). Свойства:
- **библиотека** (`source`); строгий режим (`set -euo pipefail`) включается **только при прямом запуске**, не при source — не течёт в интерактивный шелл оператора;
- токен только `GH_WRITE_TOKEN` (fail-closed), передаётся в curl через `--config` из stdin (`-K -`), **не через argv** — не виден в `ps`/`/proc/*/cmdline`; read-only ingest-токен для записи не годится;
- **privacy fail-closed**: запись в feedback-репо (`comment_issue`/`close_transferred`) идёт через guard `_assert_feedback_target` — отказ, если `FEEDBACK_REPO` не задан, ещё несёт плейсхолдер `REPLACE-ME` или равен `PROJECT_REPO` (осознанный оверрайд — `FEEDBACK_ALLOW_PUBLIC=1`; live-проверка приватности — `FEEDBACK_VERIFY_PRIVATE=1`);
- `repo`/`num` валидируются (slug-regex / `^[0-9]+$`) — defence-in-depth от path-traversal;
- каждый вызов проверяет **HTTP-код** и падает на `>=300` (не «тихий успех» при 4xx/5xx); есть `--connect-timeout`/`--max-time`;
- payload пишется во временный файл `mktemp` (`umask 077`) и удаляется **явно на всех путях** (без `trap RETURN` — он глобален и повторно стрелял бы из вызывающих функций под `set -u`);
- лейблы тримятся (`.strip()`), чтобы не плодить дубли-лейблы с пробелом;
- рассчитан на **bash** (массивы, `[[ =~ ]]`): «переносимость» = между bash-средами (Linux/macOS/WSL), не POSIX sh.

Использование (репозитории — через переменные `PROJECT_REPO`/`FEEDBACK_REPO`, дефолты в скрипте):
```bash
source scripts/feedback-triage.sh          # требует GH_WRITE_TOKEN; PROJECT_REPO/FEEDBACK_REPO переопределяемы через env
cat > /tmp/body.md <<EOF
## Проблема ...
## Источник (обратная связь, приватный репо)
- ${FEEDBACK_REPO}#43
EOF
create_issue      "$PROJECT_REPO"  "Заголовок по сути корня" /tmp/body.md "bug"
comment_issue     "$FEEDBACK_REPO" 43 "Перенесено: $PROJECT_REPO#NNN. Закрываю."
close_transferred "$FEEDBACK_REPO" 43
```

### 8.3 Валидация (без вызовов GitHub)

| «Закрой как перенесённые» | `closed / not_planned` + ссылка на проектный issue. |
| «Добавь от меня проблему …» | Завести issue от имени оператора. |

## 11. Definition of Done для итерации

- Все **открытые** отзывы: оформлены в проектные issue / сопоставлены с существующими / признаны дубль/по-дизайну.
- Разобранные отзывы **закрыты** с комментарием-связкой (двусторонняя трассируемость).
- Новые проектные issue имеют метки, источник-ссылки и связи.
- Итог подведён оператору: что заведено, что перенесено, что осталось открытым и почему.
