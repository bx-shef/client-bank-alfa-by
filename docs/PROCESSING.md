# Инструкция: обработка банковской выписки в CRM

> Last reviewed: 2026-07-05

Как приложение обрабатывает каждую операцию из выписки: какие данные берёт, какие решения принимает
и в какой последовательности, какие параметры учитывает.

По каждой операции — **сохранение/обработка** (запись операции и, если можно, распределение денег по
инвойсам/сделкам) и **оповещение** (люди узнают о движении).

> **Это целевая спецификация** (реализация поэтапно — см. [`REFACTOR_PLAN.md`](REFACTOR_PLAN.md)):
> часть механик (смарт-процессы, распределение по инвойсам/сделкам, кнопки, чат-бот) ещё **не
> реализована**; сейчас в коде — базовые `companyLookup`/`writeActivityViaRest` и стаб чата.
> Переплату/недоплату **не считаем** — просто фиксируем факт движения денег.
> Валюту везде пишем **кодом** (`BYN`, `RUB`, `CNY`, `KZT`, `USD`, `EUR`, … — ISO-код валюты), не словами.

> **Граница free / платно (#109).** Базовое **однозначное** разнесение — кандидат в **бесплатную**
> версию. **Платно** — нестандартные правила сопоставления, пограничные случаи, коробка/сервер,
> интеграции под процессы клиента. (Детальная отстройка — будущий `docs/POSITIONING.md`, пока не заведён.)
>
> **Критерий совпадения (решение владельца).** Кандидат подходит, когда у него совпадают **и сумма,
> и валюта** (строго, без переключателя). Из кандидатов заранее исключены инвойсы с **отрицательной
> стадией** и сделки с **отрицательной стадией** (в них оплаты не ищем). Дальше:
> - **точных совпадений нет** (частичная/групповая оплата — сумма не сходится) → **очередь ручного
>   разбора** (§2, §5);
> - **одно** совпадение → авторазнесение;
> - **несколько** совпадений → авторазнесение на **минимальный ID** **и оповещение в чат** (тот же
>   чат, куда шлём ошибки) — чтобы человек проверил неоднозначность.

## 1. Входные данные операции

| Поле | Роль в обработке |
|---|---|
| **Дата операции** | дата движения; период; часть ключа дедупликации |
| **Сумма** | часть ключа дедупликации; в оповещении — с кодом валюты |
| **Валюта** | ISO-код (`BYN`, `RUB`, `CNY`, `KZT`, `USD`, `EUR`, …); часть ключа дедупликации |
| **Направление** | приход / расход |
| **Расчётный счёт контрагента** | **основной ключ поиска компании** и **ключ фильтров/исключений** |
| **Наименование контрагента** | отображение; вспомогательный ключ фильтра (опц.) |
| **Наш счёт** | счёт, по которому прошла операция; **ключ поиска МОЕЙ компании** (всегда) |
| **Назначение платежа** | из него **распознаётся номер инвойса** и **идентификатор оплаты в сделке** (id сделки / поле сделки / номер заказа сделки / номер оплаты из заказа) |
| **Номер и дата документа** | идентификация; часть ключа дедупликации |
| **УНП контрагента** | **справочно, может отсутствовать**; в поиске/фильтрах не используется |

### Ключ дедупликации (усиленный)
Источников много (разные банки/форматы) — уникальности одного номера документа не гарантируем:
```
наш счёт + номер документа + дата + сумма + валюта + направление + хеш[назначение + счёт контрагента]
```
Совпал ключ ⇒ операция уже обработана (пропускаем).

> **Текущее состояние vs цель.** Сейчас реализован упрощённый ключ `account|docId`
> (`dedupKey()` в `app/utils/statement.ts`, персистентный `server/utils/activityDedupStore.ts`).
> Составной ключ выше — **целевое усиление**. Это breaking-change: **оба** слоя дедупа (in-batch `Set`
> в `handleCrmSyncJob` и персистентный стор) переводятся на новый ключ **одновременно**, иначе защита от
> дублей молча деградирует. `хеш[…]` — **отдельная чистая функция** с явной **нормализацией** входа
> (тримминг/схлопывание пробелов, регистр, юникод) и фиксированным алгоритмом (напр. SHA-256), покрытая
> тестом. Схему БД менять не нужно (`dedup_key` — TEXT). Миграция — отдельная задача.

> **Факт разнесения — отдельно от дедупа операции (#109).** Храним факт «платёж X **разнесён** на
> сущность Y» — **идемпотентно** (по ключу `платёж|сущность` дубль факта не создаём), но это **запись
> со статусом** (`разнесён` → `откат`), а **не** буквальный write-once-клон `activity_dedup`: **сторно**
> платежа переводит статус в `откат` (или ручной разбор), запись **не стирается**. Переживает
> **рестарт** и **переимпорт** (повторная выписка не разнесёт дважды). Скоуп по `member_id` (per-portal).
> Отдельный стор от дедупа операции.

## 2. Последовательность принятия решений

### Этап A. Входной контроль
```
Операция
  ├─[A1] Данные валидны? (сумма, дата, наш счёт, направление, валюта) — нет ──► ОШИБКА (§5)
  ├─[A2] Дубликат по составному ключу? — да ──► ПРОПУСТИТЬ
  ├─[A3] Контрагент в исключениях? (по расчётному счёту; опц. по названию) — да ──► НЕ ЗАГРУЖАТЬ
  └─ иначе ──► Этап B
```

### Этап B. Направление
Определяем направление: **приход** (наш счёт — получатель) / **расход** (наш счёт — плательщик).
Направление **фиксируется** (часть ключа дедупликации, хранится в операции), но логику **поиска и
фиксации не меняет** — движение денег обрабатывается одинаково. Отражается в тексте оповещения.

### Этап C. Поиск компаний (обе — по расчётному счёту)
1. **Моя компания** — **всегда** ищем по **нашему** расчётному счёту (наш счёт из операции).
   Определяет мою компанию движения → ответственного и привязку записи. **Не найдена ⇒ запись в
   чат с ошибками** (операцию корректно не привязать; §5).
2. **Компания-клиент (контрагент)** — по расчётному счёту контрагента.
   - Не найдена ⇒ **UNMATCHED-клиент**: **дело в мою компанию** (из п.1) + **оповещение в чат
     ответственного за мою компанию** (§5), чтобы проблему точно заметили.

### Этап D. Сохранение и обработка — ветвление по тарифу
```
Тариф поддерживает смарт-процессы?
│
├─ ДА:
│   1) Сохранить ЭЛЕМЕНТ смарт-процесса
│        привязка: к МОЕЙ компании + к компании-КЛИЕНТУ
│        (в карточке клиента — закладка с платежами)
│   2) Настройка «распределять автоматически» включена?
│        ├─ ДА → искать инвойсы и оплаты сделок → РАСПРЕДЕЛИТЬ (см. ниже)
│        └─ НЕТ → элемент сохранён; распределение ВРУЧНУЮ позже (§3)
│
└─ НЕТ (тариф не тянет смарт-процессы):
    · сохранить ДЕЛО в таймлайне КОМПАНИИ-КЛИЕНТА (если найдена),
      иначе — в таймлайне МОЕЙ компании + текст с ПРИЧИНОЙ («клиент не определён …»)
    · распределения нет — только запись + оповещение
    · показать спокойную подсказку: «распределение оплат доступно на тарифе со смарт-процессами»
```

**Подсказка про смарт-процессы** (только на тарифе без смарт-процессов) — аккуратная карточка с
иконкой: поясняет, что распределение оплат по инвойсам/сделкам доступно на тарифе со смарт-процессами
(образец вида — b24ui Card
[«sales-dynamics-widget»](https://bitrix24.github.io/b24ui/raw/docs/components/card.md#sales-dynamics-widget)).
Тон — **профессиональный, без давления** (акцентный цвет допустим, «настырность» не нужна: наш клиент
и так понимает ценность процессов — см. ICP в [`POSITIONING.md`](POSITIONING.md)). По клику —
пояснение про апгрейд и кнопка «Создать смарт-процесс».

**Распределить** (когда найдены инвойс/оплата). Поиск связанных **инвойсов и оплат сделок ведётся
в рамках МОЕЙ компании + компании-клиента** — иначе при нескольких юрлицах будет путаница.
Ключи берём из **назначения платежа**: **инвойс** — по распознанному номеру инвойса; **оплату в сделке**
— по распознанному идентификатору (id сделки / поле сделки / номер заказа / номер оплаты из заказа).

**Единое ядро разнесения (#109).** Целевые сущности — **оплата сделки** и **счёт-инвойс** (различаются
конечная сущность и действие «отметить оплату»); «заказ» — не отдельная цель, а **способ найти оплату
сделки** (по номеру заказа/оплаты, §4). Проектируем **одним** ядром (чистое, DI, тесты) с абстракцией
над целью, а не копиями. Ядро принимает **уже отфильтрованный по компаниям** кандидат (ID из Этапа C/D),
а **не** делает собственный lookup по произвольному ID — чтобы не тронуть чужую сущность.

**Фильтр по стадии (до подбора кандидатов).** Инвойсы с **отрицательной** стадией (провал/отмена)
исключаем; сделки с **отрицательной** стадией тоже — в них оплаты **не ищем**. То есть в кандидаты
попадают только сущности из «живых» стадий.

**Что выбираем:**
- **Сумма и валюта обязаны совпадать** (решение владельца, без переключателя): инвойс или оплату
  сделки с суммой или валютой, **отличной** от платежа, **выбрать нельзя** — исключаем из кандидатов;
- найдены **и инвойс, и оплата**, причём **инвойс связан с этой сделкой** (`parentId`) — это одна цель
  в двух представлениях → предпочтение **инвойсу**; **если связи нет** — это **разные** кандидаты;
- **нет ни одного точного совпадения** (частичная/групповая оплата — сумма не сходится) → сущность
  **не трогаем**, кладём в **«очередь ручного разбора»** и уведомляем;
- **одно** совпадение → авторазнесение;
- **несколько разных совпадений** → авторазнесение на **минимальный ID** **и оповещение в чат** (тот
  же чат, куда шлём ошибки), чтобы человек проверил неоднозначность. (Полные дубли одной сущности —
  тот же ID — схлопываются молча, это не неоднозначность.)

**Порядок и ветки поиска (при включённом авторазнесении — а значит, есть смарт-процессы).** Сначала
ищем **инвойс**; **не нашли инвойс → ищем среди оплат сделок**. Итог:
- **найдено** → размечаем по «Что выбираем» выше (одно → разносим; несколько → минимальный ID +
  оповещение в чат);
- **не найдены ни инвойс, ни оплаты** → **дело в компанию клиента** + оповещение **ответственному за
  клиента** «**некуда разнести**»; сумма остаётся в элементе смарт-процесса (поле «осталось
  распределить»), пользователь распределяет **вручную**.

**Авторазнесение выключено** → авто-поиск **не запускаем**, ветки «не найдено» не возникают;
распределение — вручную из элемента (§3). Кнопка **«разнести автоматически»** из элемента запускает
**тот же** поиск для этого элемента, но **новый элемент не создаём** (он уже существует).

**«Очередь ручного разбора»** — не новый открытый эндпоинт: это **состояние** элемента смарт-процесса
(поле **«осталось распределить»** > 0 — **стадий у элемента нет**) / дела, под **нативными правами
CRM** Б24, а не отдельная незащищённая выборка.

**Действия:**
- **Сделка** — стадию **напрямую не меняем**. Отметка **«оплачено»** в оплате сделки
  ([crm.item.payment.pay](https://apidocs.bitrix24.ru/api-reference/crm/universal/payment/crm-item-payment-pay.html)
  — меняет **только статус оплаты**, доп. полей не принимает) + **отдельный вызов триггера**
  (сумма/валюта/назначение передаём через него, не через `payment.pay`).
- **Инвойс** — стадия по настройке (указана → перевести; не указана → не трогаем) и/или **триггер в инвойсе**.

**Суть триггера:** сигнал «**пришли деньги**» (по возможности — с суммой и валютой). Решение «вся сумма
или часть», «сюда или в соседний инвойс» **клиент принимает сам** через свои бизнес-процессы (БП). Если
REST-триггер позволяет передать доп. параметры (сумма/валюта/назначение) — **передаём их**
(см. [документацию триггеров](https://apidocs.bitrix24.ru/api-reference/crm/automation/triggers/index.html)).

Смарт-процесс — пользовательский объект:
[user-defined object types](https://apidocs.bitrix24.ru/api-reference/crm/universal/user-defined-object-types/index.html).
Привязка элемента к двум компаниям (моя + клиент) — **отдельными полями элемента**, не нативной
мульти-привязкой. **У элемента стадий НЕТ** — вместо стадии есть **read-only поле «осталось
распределить»** (сколько денег ещё не разнесено; редактировать нельзя, считает приложение). **Переход тарифа**: кнопка «Создать смарт-процесс» при апгрейде; если тариф не
поддерживает — уведомление, что смарт-процесс недоступен. **Апгрейд** (появились смарт-процессы):
уже созданные **дела остаются как есть** (ничего не мигрируем — по аналогии: при **даунгрейде** мы не
удаляем накопленные элементы смарт-процесса), **новые** операции идут в смарт-процесс.

### Этап E. Оповещение (в обеих ветках)
1. **Дело** — конфигурируемое дело в таймлайне, с кнопками (§6). В ветке без смарт-процессов это
   и есть основная запись об операции.
   [configurable timeline activity](https://apidocs.bitrix24.ru/api-reference/crm/timeline/activities/configurable/structure/layout.html)
2. **Сообщение чат-бота** в выбранный чат (в настройке: **какой чат** + **что писать** —
   приходы / расходы / всё). Тексты (сумма — с кодом валюты):
   - **оплата по сделке**: дата · контрагент · `сумма BYN` · основание · **ссылка на сделку** ·
     ответственный за сделку;
   - **инвойс**: дата · контрагент · `сумма BYN` · основание · **ссылка на инвойс** ·
     ответственный за инвойс.

## 3. Распределение денег (в смарт-процессе)

Режим задаёт настройка **«распределять автоматически»** (доступна **только при поддержке
смарт-процессов**):
- **включена** → приложение разносит деньги **автоматически**; ручной разнос **заблокирован**
  (кнопки ручного разноса показывают сообщение, что включён авто-режим);
- **выключена** → разносим **вручную** из элемента; дополнительно есть кнопка **`разнести автоматически`**
  — разово запускает то же действие, что и авто-режим, для этого элемента.

Эта же настройка **включает/выключает работу триггера**: включена → триггер срабатывает (при разносе);
выключена → автоматически триггер **не** срабатывает (только при ручном разносе).

Кастомные поля-**кнопки** в элементе смарт-процесса (мы их создаём):

- **`повторно поискать клиента`** — деньги пришли, а клиент не найден: пользователь вручную вносит
  **расчётный счёт** (в карточку клиента / реквизиты), затем жмёт кнопку в элементе → приложение
  **заново ищет компанию** по счёту, находит и (при возможности) распределяет.
- **`поискать что оплачено`** — распределить деньги по инвойсам/оплатам вручную (при **выключенном**
  авто-разносе):
  1. берём **элемент** (операцию) и его **сумму**;
  2. ищем **инвойсы/оплаты** для покрытия — **в рамках МОЕЙ компании и компании-клиента**
     (при нескольких юрлицах иначе будет путаница);
  3. **кладём деньги**: оплата → «оплачено», инвойс → оплаченная стадия.
- **`разнести автоматически`** — доступна при **выключенной** галке; разово запускает авто-разнос
  для этого элемента.

Если клиент или моя компания не определены — по нажатию кнопки пользователь сразу увидит проблему.

### Реакция на изменения после распределения
- **Авто-распределение** — последующая **смена стадии инвойса** или **снятие оплаты** **никак не влияет**
  на элемент смарт-процесса (повторно ничего не делаем).
- **Ручное распределение** — **смена стадии инвойса** или **снятие оплаты** → элемент **требуется
  распределить снова** (помечаем как «требует распределения», пользователь разносит заново).

## 4. Параметры, которые учитывает приложение

- **Распознавание из назначения платежа** — фраза-триггер (число после неё = идентификатор).
  Распознаём **номер инвойса** и **идентификатор оплаты в сделке** (id сделки / поле сделки /
  номер заказа / номер оплаты из заказа).
- **Распределять автоматически** — вкл/выкл (только при смарт-процессах). Вкл → авто-разнос, ручной
  заблокирован; выкл → ручной разнос + кнопка «разнести автоматически» (§3). Также **вкл/выкл работу
  триггера** (§2 Этап D).
- **Стадия инвойса при оплате** — указать стадию (перевести в неё) или **не указывать** (стадию не
  трогаем); смена также возможна **триггером в инвойсе**. Для **сделки** стадию не задаём — только
  отметка «оплачено» + триггер.
- **Уведомления** — какой чат; что писать (приходы / расходы / всё).
- **Получатель ошибок** — пользователь (из настроек), которому чат-бот шлёт ошибки/проблемы (§5).
- **Фильтры-исключения** — список **расчётных счетов** (опц. названий) контрагентов: не загружать /
  не уведомлять.
- **Хранение** — смарт-процесс (если тариф поддерживает) либо список дел; кнопка перехода при апгрейде.
- **Кнопки в деле** — набор действий (§6).

## 5. Ошибки, UNMATCHED и ручная корректировка

Ошибки и проблемы обработки **чат-бот шлёт указанному в настройках пользователю** (§4 «Получатель ошибок»).

- **Ошибка файла/данных** — оповестить получателя ошибок; при необходимости — **создать issue**.
  ⚠️ **Банковская тайна.** Файл выписки (счета, суммы, контрагенты) в **публичный** issue по умолчанию
  **не прикладываем**. Варианты по возрастанию: (1) маскированный фрагмент (счета/суммы частично
  скрыты — хватает для диагностики парсера); (2) приватный канал (приватный трекер / почта с
  ограниченным доступом) для полного файла; (3) публичный issue с файлом — **только по явному
  активному подтверждению** («да, понимаю, файл станет публичным»). Канон «без PII по умолчанию» —
  как в [`REFACTOR_PLAN.md`](REFACTOR_PLAN.md) (#61).
- **UNMATCHED — клиент не определён** (компания-контрагент не найдена по счёту) — дело **дублируется**
  и привязывается к **моей компании** (из §2 Этап C.1), ответственный — **ответственный за мою
  компанию** — так проблема гарантированно замечена.
- **Моя компания не определена** (не нашли по нашему счёту — §2 Этап C.1) — привязать дело/элемент
  не к чему: **в CRM не пишем**, шлём проблему **получателю ошибок** (чат-бот) с причиной («наш счёт
  … не сопоставлен ни с одной моей компанией»); оператор заводит реквизит и повторяет обработку.
- **Структура смарт-процесса повреждена/удалена** (админ удалил элемент или поле, которое должно
  быть): ловим «не хватает того, что должно быть» → пишем проблему **в чат админу**, **пересоздаём
  структуру** и **повторяем** обработку операции. Если со **2-й** попытки снова ошибка — пишем проблему
  **в чат админу** и дальше **не зацикливаемся**.
- **Ручная корректировка** — на **деле** или в **смарт-процессе**. При изменении привязок
  **ничего автоматически не пересчитывается** — пересчёт запускает **сам пользователь** кнопками
  элемента (§3: `повторно поискать клиента` / `поискать что оплачено` / `разнести автоматически`).

## 8. Открытые вопросы (к стадии реализации)

Не решаются этим документом — фиксируются как кандидаты в issue перед реализацией:

- **Фразы-триггеры распознавания** назначения платежа — таблица «фраза/regex → тип идентификатора»
  (инвойс / id сделки / поле / номер заказа / номер оплаты) + **5–10 реальных строк назначения** как
  фикстуры (по образцу `tests/fixtures/client-bank/`).
- **Обнаружение изменений** (§3: смена стадии инвойса / снятие оплаты → «требует распределения») —
  транспорт события: вебхук `ONCRMDYNAMICITEMUPDATE` / подписка на поле / поллинг.
- **Определение тарифа** (поддержка смарт-процессов) — метод REST (напр. проба `crm.type.list`).
- **`unp` пусто vs undefined** — тип `StatementParty.unp: string` обязателен; зафиксировать конвенцию
  (пустая строка = «нет УНП»), чтобы фильтры/отображение не разошлись.
- **Rate-limit REST** — на операцию теперь 2 поиска компании + поиск инвойсов/оплат; учесть рост
  вызовов до реального объёма (связать с #90/#91).
- ✅ **Критерий совпадения и поведение при неоднозначности — решены:** кандидат = совпали сумма и
  валюта (строго, без переключателя); из кандидатов исключены инвойсы/сделки с отрицательной стадией.
  Нет точного совпадения → ручной разбор; одно → авторазнесение; несколько → минимальный ID **+
  оповещение в чат**. См. врезку в преамбуле и §2.
