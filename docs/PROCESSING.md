# Инструкция: обработка банковской выписки в CRM

> Last reviewed: 2026-07-17

Как приложение обрабатывает каждую операцию из выписки: какие данные берёт, какие решения принимает
и в какой последовательности, какие параметры учитывает.

По каждой операции — **сохранение/обработка** (запись операции и, если можно, распределение денег по
инвойсам/сделкам) и **оповещение** (люди узнают о движении).

> **Это целевая спецификация** (реализация поэтапно — см. [`REFACTOR_PLAN.md`](REFACTOR_PLAN.md)):
> часть механик (смарт-процессы, распределение по инвойсам/сделкам, кнопки, чат-бот) ещё **не
> реализована**; сейчас в коде — базовые `companyLookup`/`writeConfigurableActivityViaRest` и стаб чата.
> Переплату/недоплату **не считаем** — просто фиксируем факт движения денег.
> Валюту везде пишем **кодом** (`BYN`, `RUB`, `CNY`, `KZT`, `USD`, `EUR`, … — ISO-код валюты), не словами.
>
> **Где живут данные (решение #109/#259).** Сами CRM-сущности (операции/сделки/инвойсы) у себя не храним —
> источник правды B24. Идемпотентность тоже **в B24** по **нашему** маркеру: операцию пишем **настраиваемым
> делом** (`crm.activity.configurable.add`) с маркером `originatorId`+`originId`, перед записью ищем его
> (`crm.activity.list`); инвойс/смарт-процесс несут `xmlId`. Стора дедупа дел у нас **больше нет** (#259).
> Итого БД держит `portal_tokens` **плюс** служебные сторы (`allocation_fact`/`import_result`/`metrics_counter`),
> не реплику CRM. Настройки — в
> `app.option` (B24). Подробно и план снятия сторов — §1 «Ключ операции и дедуп».

> **Граница free / платно (#109, решение принято).** **Бесплатно** — импорт и **разнесение оплат**
> (базовое). **Платно** — **кастомная разработка** под процессы клиента и **перенос на его сервер**;
> нестандартные правила сопоставления и пограничные случаи. Распределение по инвойсам/сделкам живёт
> в смарт-процессе (тариф Bitrix24): клиент на бесплатном тарифе — **не целевой**, ему показываем
> баннер про апгрейд тарифа; **всем** показываем баннер про кастом и перенос на сервер (§2 Этап D).
> Детальная отстройка — [`POSITIONING.md`](POSITIONING.md).
>
> **Критерий совпадения (решение владельца).** Кандидат подходит, когда у него совпадают **и сумма,
> и валюта** (строго, без переключателя). Из кандидатов заранее исключены инвойсы с **отрицательной
> стадией** и сделки с **отрицательной стадией** (в них оплаты не ищем). Дальше:
> - **точных совпадений нет** (частичная/групповая оплата — сумма не сходится) → **очередь ручного
>   разбора** (§2, §5);
> - **одно** совпадение → авторазнесение;
> - **несколько** совпадений → авторазнесение на **минимальный ID** **и оповещение в чат** (тот же
>   чат, куда шлём ошибки) — чтобы человек проверил неоднозначность.

## 0. Путь данных: приём → очередь → внесение

Один конвейер обрабатывает выписку из **онлайн-банка** и из **ручной загрузки файла** — различается
только источник. Ниже — путь **ручного импорта**; онлайн-опрос отличается лишь шагами 1–4 (вместо
файла — забор окна выписки у банка по расписанию). Инфраструктура очередей и **как наращивать
обработчики под нагрузку** — [`QUEUES.md`](QUEUES.md).

**Приём (браузер → сервер).**
1. На `/import` браузер разбирает файл **локально** и показывает предпросмотр — данные никуда не уходят.
2. По кнопке «Записать в CRM» на сервер уходит **сам файл** (не разобранные операции): единственный
   авторитет разбора — на сервере, браузер не может подменить операции.
3. Эндпоинт приёма аутентифицируется **фрейм-токеном** (`Authorization: Bearer <access_token>` +
   `X-B24-Domain`, как `/api/chat-settings`). `member_id` из UI **не доверяем** — резолвим **на сервере
   проверенно**: вызываем лёгкий REST этим же токеном (напр. `profile`/`app.info`), ответ авторитетно
   даёт **портал** (`member_id`) и **пользователя-инициатора**. Инициатор — это тот, от чьего имени идёт
   импорт: записываем его автором записи и используем там, где B24 нужен «кто внёс». **Проверка ключа
   портала** (как на входе воркера, см. «Правила доставки»): ключа нет — приложение уже удалено — файл
   **не принимаем**, пакет не создаём.

**Очередь разбора → очередь внесения.**
4. Сервер кладёт файл в **пакет разбора** (`file-parse`). Файл едет **в пакете** (лимит размера уже
   есть) — отдельного хранилища файла у нас нет.
5. Воркер разбора декодирует (windows-1251) и парсит файл → `StatementItem[]`, формирует **пакет
   операций** и кладёт его в очередь **внесения** (`crm-sync`). В пакете помечен **портал**.
6. Воркер внесения берёт пакет и идёт по операциям — логика одной операции в §2.

**Правила доставки (надёжность + изоляция).**
- **Нет ключа портала → брак.** Воркер, взяв пакет, **первым делом** проверяет ключ портала. Нет
  (портал удалён) → пакет **отвергаем**: убираем из очереди (не повторяем) + строка в лог «пакет из
  B24 `<домен>` не нашёл свой ключ — отвергнут». Та же проверка — на приёме файла (шаг 3). Это **не**
  временная ошибка: ключ пропал насовсем, повтор бессмыслен.
- **Временная ошибка → повтор.** Сеть / лимит / сбой — пакет **повторяем** с задержкой (at-least-once).
- **Повторная запись невозможна.** Идемпотентность: носитель с нашим маркером (настраиваемое дело —
  `originatorId`+`originId` / элемент СП — `xmlId`) → поиск метки в B24 перед записью; нашли → пропускаем.
  Стора дедупа у нас нет — маркер живёт на самой записи в B24 (§1).
- **Порталы не смешиваются.** В каждом пакете — свой `member_id`; воркер ходит в REST **только** ключом
  этого портала (в чужой портал дотянуться нельзя); все выборки скоуплены `member_id`.
- **Наша БД — не реплика CRM.** Сами операции/сделки/инвойсы и настройки живут **в B24** (§1, §4). Postgres
  держит `portal_tokens` **плюс** служебные сторы (`allocation_fact` — факт разнесения, `import_result`/
  `metrics_counter` — статус/метрики); дедуп дел живёт в B24 (маркер), стора у нас нет (§1).

## 1. Входные данные операции

| Поле | Роль в обработке |
|---|---|
| **Дата операции** | дата движения; период; часть ключа дедупликации |
| **Сумма** | часть ключа дедупликации; в оповещении — с кодом валюты |
| **Валюта** | ISO-код (`BYN`, `RUB`, `CNY`, `KZT`, `USD`, `EUR`, …); часть ключа дедупликации |
| **Направление** | приход / расход |
| **Расчётный счёт контрагента** | **основной ключ поиска компании** и **ключ фильтров/исключений** |
| **Наименование контрагента** | отображение; вспомогательный ключ фильтра (опц.) |
| **Наш счёт** | счёт, по которому прошла операция; **ключ поиска МОЕЙ компании** (всегда) |
| **Назначение платежа** | из него распознаётся **идентификатор** сущности (инвойс / сделка / заказ / оплата заказа / смарт-процесс / номер документа генерации → привязанная сущность) — полный перечень источников номера в §4 |
| **Номер и дата документа** | идентификация; часть ключа дедупликации |
| **УНП контрагента** | **справочно, может отсутствовать**; в поиске/фильтрах не используется |

### Ключ операции и дедуп — через метку в B24 (не в нашей БД)
Источников много (разные банки/форматы) — уникальности одного номера документа не гарантируем:
```
наш счёт + номер документа + дата + сумма + валюта + направление + хеш[назначение + счёт контрагента]
```
`хеш[…]` — **отдельная чистая функция** с явной **нормализацией** входа (тримминг/схлопывание пробелов,
регистр, юникод) и фиксированным алгоритмом (напр. SHA-256), покрытая тестом.

**Где живёт дедуп (решение #109, уточнено #259).** Идемпотентность стремимся держать **не у себя**, а в
самом Bitrix24 — но это возможно **только для сущностей с искомым маркером**. Live-проверка полей на
тест-портале (#259):

| Куда пишем | id / семейство | Маркер внешней привязки | Ищется (`*.list`) |
|---|---|---|---|
| Сделка / лид / контакт / компания | `crm.item` 2/1/3/4 | `originId` + `originatorId` | да (`crm.item.list`) |
| Смарт-счёт (инвойс) | `crm.item` 31 | `xmlId` | да (`crm.item.list`) |
| Смарт-процесс (СПА) | `crm.item` ≥1030 | `xmlId` | да (`crm.item.list`) |
| **Простое** дело (`crm.activity.todo.add`) | `crm.activity` | **нет** (метод не принимает) | нет |
| **Настраиваемое** дело (`crm.activity.configurable.add`) | `crm.activity` | **`originId` + `originatorId`** (метод принимает) | **да** (`crm.activity.list`) — ⚠ только в контексте приложения (OAuth) |
| КП (предложение) | `crm.item` 7 | нет | нет |

**Чей маркер — важнее, чем какой.** `originId`/`xmlId` на **клиентской** сделке/инвойсе (строки 1–3) —
**чужие**: клиент использует их под свою интеграцию, мы их **не штампуем** (см. ниже). А `originId` на
**нашем настраиваемом деле** и `xmlId` на **нашем элементе смарт-процесса** — **наши** поля на **наших**
записях → **пишем** свободно (без штамповки чужого). Значит B24-дедуп для операции даёт **настраиваемое
дело** (`originId`, `crm.activity.configurable.add`) **или** **элемент смарт-процесса** (`xmlId`) — оба
создаём мы; у **простого** дела маркера нет вовсе. *(Поправка к #9/#259: `todo.add` без маркера — верно, но
`configurable.add` его несёт; #9 рассмотрел только `todo.add`.)*

**Ищем дело строго по паре `ORIGINATOR_ID` + `ORIGIN_ID`.** `crm.activity.list` вернёт **любое** дело портала
с данным `ORIGIN_ID` (чужого провайдера / импорта), поэтому `ORIGINATOR_ID` обязан быть нашим **отличимым
app-namespace** — фильтр по одному `ORIGIN_ID` даст **ложный дедуп** (легитимно-новую операцию молча
пропустим). Это правило действует **везде**, где ниже упомянут «поиск настраиваемого дела».

Отсюда модель:
- **запись операции — настраиваемым делом** (`crm.activity.configurable.add`, `originatorId`+`originId`=ключ
  операции); **перед записью** ищем ключ (`crm.activity.list filter[ORIGINATOR_ID][ORIGIN_ID]`) → нашли →
  операция уже внесена, **пропускаем**. Маркер пишется **атомарно** с делом → нет write→remember-зазора и
  стора дедупа не нужно. Инвойс/смарт-процесс несут `xmlId` (`crm.item.list filter[xmlId]`) — для будущего
  носителя-элемента СП (§2/§5). Все носители создаём мы; идемпотентность и переживание рестарта/переимпорта
  обеспечивает B24;
- **факт разнесения** для проведённой цели — это **состояние самого объекта**, которое мы **и так читаем**:
  для `deal-payment` — детерминированно (`payment.pay`→`paid='Y'`, `paymentLookup` его отсеивает); для
  **инвойса — только если** настроенная `allocation.invoicePaidStageId` указывает на стадию с `SEMANTICS='S'`
  (иначе `negativeStages` её не исключит и цель пере-мутируется на редоставке). Именно поэтому пре-чек
  мутации читает **состояние цели в B24** (`isTargetApplied`, а не факт) — **Фаза A (amount-путь) сделана**;
- **чужие поля не штампуем.** `originId`/`xmlId` на **клиентской** сделке/инвойсе (строки 1–3 матрицы) клиент
  может использовать под свою интеграцию — перезапись инвазивна. Наш маркер — только на **нашей** записи
  (настраиваемое дело / элемент СП). В дефолтном `autoDistribute=OFF` (мутации нет → следа в B24 нет) снятие
  `allocation_fact` вяжется к носителю с нашим маркером, а не к штамповке чужого поля;
- метка строится из **составного** ключа выше: слабый `account|docId` → дедуп по метке унаследует слабость
  (коллизии/потеря операции). Усиление ключа **питает метку** (§1 follow-up).

Точные **поле метки** и **метод/фильтр поиска** (какой `*.list`) фиксируются на REST-слайсе в
[`REST_METHODS.md`](REST_METHODS.md) — как имена полей сущностей в §4.

> **Текущее состояние.** Операцию crm-sync пишет **настраиваемым делом** — дедуп в B24 (маркер), **стора
> `activity_dedup` больше нет** (#259: `app/utils/configurableActivity.ts` + `server/utils/configurableActivityWrite.ts`
> + `server/utils/activityMarkerLookup.ts`; маркер `originatorId`+`originId`, поиск `crm.activity.list`;
> `rememberActivity` ушёл — маркер атомарен с делом). Из дедуп-сторов остался только `allocationFactStore.ts`
> (`allocation_fact`, факт разнесения). Ключ операции пока **упрощённый** `account|docId` (`dedupKey()`).
> - **Фаза A (amount-путь — сделана):** идемпотентность мутации разнесения читает **состояние цели в B24**, а не
>   `allocation_fact`: `isTargetApplied` (`server/utils/allocationApplied.ts` `readAllocationApplied`) — `deal-payment`:
>   `paid='Y'` (`crm.item.payment.list`); инвойс: уже на `invoicePaidStageId` (`crm.item.list`). Это **точнее** факта

> - **Ключ операции** — усиление до составного хеша (§1) — ещё нет.
> - **Элемент смарт-процесса** как носитель (`xmlId`, §2/§5) — альтернатива настраиваемому делу на платном
>   тарифе (плюс кнопки §6); ещё не реализован.

>   убедиться, что app-пользователь видит свои дела;
> - **+1 REST на операцию:** дедуп-поиск идёт на **каждую** уникальную операцию (до поиска компании) —
>   учитывать в REST-бюджете (#191, SDK-лимитер).

## 2. Последовательность принятия решений

### Этап A. Входной контроль
```
Операция
  ├─[A1] Данные валидны? (сумма, дата, наш счёт, направление, валюта) — нет ──► ОШИБКА (§5)
  ├─[A2] Контрагент в исключениях? (по расчётному счёту; опц. по названию) — да ──► НЕ ЗАГРУЖАТЬ
  ├─[A3] Уже внесена? (маркер `ORIGINATOR_ID`+`ORIGIN_ID` настраиваемого дела → поиск `crm.activity.list`; будущий носитель-элемент СП — по `xmlId`, #259) — да ──► ПРОПУСТИТЬ
  └─ иначе ──► Этап B
```
**Порядок A2 → A3 (исключение раньше дедупа).** Исключение проверяется по **локальному** списку из
настроек (дёшево), а дедуп — это **REST-поиск** в B24; дешёвую проверку делаем первой, чтобы не тратить
вызов на операцию, которую всё равно не грузим. Ещё раньше, до разбора операций, отсекается **пакет без
ключа портала** (§0 «Правила доставки»).

### Этап B. Направление
Определяем направление: **приход** (наш счёт — получатель) / **расход** (наш счёт — плательщик).
Направление **фиксируется** (часть ключа дедупликации, хранится в операции), но логику **поиска и
фиксации не меняет** — движение денег обрабатывается одинаково. Отражается в тексте оповещения.

### Этап C. Поиск компаний (обе — по расчётному счёту)
1. **Моя компания** — **всегда** ищем по **нашему** расчётному счёту (наш счёт из операции).
   Определяет мою компанию движения → ответственного и привязку записи. **Не найдена ⇒ запись в
   чат с ошибками** (операцию корректно не привязать; §5).
2. **Компания-клиент (контрагент)** — по расчётному счёту контрагента.
   - Не найдена ⇒ **UNMATCHED-клиент**: запись **в мою компанию** (из п.1; на тарифе со смарт-процессами —
     элемент со связью только с моей, без — дело в её таймлайн), **ответственным** остаётся ответственный
     за мою компанию, а **проблема-оповещение** идёт в **чат ошибок** (§5), чтобы её точно заметили.

> **Текущее состояние (v1 ручного импорта).** В коде `crm-sync` пока делает упрощённый путь стадии 4:
> клиент найден → **дело** в его таймлайн; **клиент НЕ найден → операция помечается `unmatched` и
> НИЧЕГО не пишется** (нет владельца для дела). Каскад «в мою компанию / элемент смарт-процесса» (эта
> ветка) — это цель #109. Поведение осознанное для первого рабочего импорта; отражено комментарием в
> `server/queue/handlers.ts` (`handleCrmSyncJob`).

### Этап D. Сохранение и обработка — ветвление по тарифу
```
Тариф поддерживает смарт-процессы?
│
├─ ДА:
│   1) Сохранить ЭЛЕМЕНТ смарт-процесса
│        привязка: к МОЕЙ компании + к компании-КЛИЕНТУ
│        (в карточке клиента — закладка с платежами)
│   2) Настройка «распределять автоматически» включена?
│        ├─ ДА → искать инвойсы и оплаты сделок → РАСПРЕДЕЛИТЬ (см. ниже)
│        └─ НЕТ → элемент сохранён; распределение ВРУЧНУЮ позже (§3)
│
└─ НЕТ (тариф не тянет смарт-процессы):
    · сохранить ДЕЛО в таймлайне КОМПАНИИ-КЛИЕНТА (если найдена),
      иначе — в таймлайне МОЕЙ компании + текст с ПРИЧИНОЙ («клиент не определён …»)
    · распределения нет — только запись + оповещение
    · показать спокойную подсказку: «распределение оплат доступно на тарифе со смарт-процессами»
```

> **Структура смарт-процесса (схема + провижининг) — отдельный слайс.** Конкретные `entityTypeId`,
> поля элемента (моя компания / клиент / read-only «осталось распределить» / кнопки §3) и **когда/как их
> создавать** (при установке / при первом использовании) + рутина **`ensure-structure`** (проверка и
> самолечение — §5) описываются на REST-слайсе; образец создания структуры — seed-скрипт (#157). Здесь —
> модель ветвления, не схема БД смарт-процесса.
> **Определение тарифа** (тянет ли смарт-процессы) — открытый вопрос §8 (проба `crm.type.list`).

**Подсказка про смарт-процессы** (только на тарифе без смарт-процессов) — аккуратная карточка с
иконкой: поясняет, что распределение оплат по инвойсам/сделкам доступно на тарифе со смарт-процессами
(образец вида — b24ui Card
[«sales-dynamics-widget»](https://bitrix24.github.io/b24ui/raw/docs/components/card.md#sales-dynamics-widget)).
Тон — **профессиональный, без давления** (акцентный цвет допустим, «настырность» не нужна: наш клиент
и так понимает ценность процессов — см. ICP в [`POSITIONING.md`](POSITIONING.md)). По клику —
пояснение про апгрейд и кнопка «Создать смарт-процесс».

**Баннер про кастом и перенос на сервер** (показываем **всем**, независимо от тарифа) — аккуратная
карточка: предлагаем **кастомную разработку** под процессы клиента и **перенос решения на его сервер**
(за деньги — это наш доход, а не апселл Bitrix24). Тон такой же профессиональный. По клику — заявка
(та же форма/канал, что на лендинге).

**Распределить** (когда найдены инвойс/оплата). Поиск связанных **инвойсов и оплат сделок ведётся
в рамках МОЕЙ компании + компании-клиента** — иначе при нескольких юрлицах будет путаница.
Ключи берём из **назначения платежа** (полный перечень источников номера — §4).

**Целевые сущности (что может закрыть платёж — приход ИЛИ расход):**
1. **Счёт-инвойс** (Smart Invoice, `entityTypeId=31`) — действие: отметка оплаты / стадия.
2. **Оплата сделки** (оплата заказа, привязанного к сделке) — действие: `payment.pay` + триггер.
3. **Сделка без оплат** (`entityTypeId=2`) — платёж привязывается к сделке напрямую; действие — **триггер**
   (объекта оплаты нет).
4. **Элемент смарт-процесса** (по сути «сделка без оплат») — действие: **триггер**/запись.

«Заказ» — не отдельная цель, а способ **найти оплату сделки** (по номеру/ID заказа). **Документ модуля
«Генерация документов»** — не цель, а **мост**: по номеру документа находим сам документ
(`crm.documentgenerator.document.list`), а из его `entityTypeId`/`entityId` — привязанную сущность
(инвойс/сделку/заказ/смарт-процесс), которую и разносим.

**Единое ядро разнесения (#109).** Проектируем **одним** ядром (чистое, DI, тесты) с абстракцией над
целью, а не копиями. Ядро принимает **уже отфильтрованный по компаниям и стадии** кандидат (ID из Этапа
C/D), а **не** делает собственный lookup по произвольному ID — чтобы не тронуть чужую сущность.

**Одна процедура на два входа.** Авто-разнесение при импорте и кнопка **«разнести автоматически»** на
элементе (§3) — это **один и тот же код**. Различается только вход: при импорте сначала создаём элемент
и зовём процедуру; кнопка зовёт её на **уже существующем** элементе (новый не создаём). Значит логику
поиска+решения+действия пишем **единожды** и переиспользуем из обоих мест.

**Два разных пути по типу цели:**
- **Инвойс / оплата сделки** — находятся **поиском** (у контрагента может быть несколько сущностей на
  одну сумму) → проходят через amount-based ядро (сумма+валюта, min-ID+`ambiguous`, §«Что выбираем»).
- **Сделка без оплат / элемент смарт-процесса** — находятся **прямой ссылкой** (id/поле из назначения) →
  это **безусловная** триггер-цель: «нашли → триггерим», **без** amount-фильтра ядра (у «голой»
  сделки/элемента нет объекта оплаты с суммой инвойса — искусственно сопоставлять сумму не нужно).

**Безопасность прямого поиска по ID (`by-id`).** Значение идентификатора берётся из **назначения
платежа**, которое контролирует **плательщик**. Поэтому результат прямой выборки по ID (сделка/инвойс/
элемент смарт-процесса) **обязан повторно пройти фильтр компаний + стадии** до любого действия — иначе
плательщик, подставив чужой ID в назначение, закроет чужую сущность (IDOR). Имя поля для поиска по
настраиваемому полю (`deal-field`/`smart-field`) **валидируем** против реального списка полей сущности,
не подставляем «как в настройках».

**Фильтр по стадии (до подбора кандидатов).** Инвойсы с **отрицательной** стадией (провал/отмена)
исключаем; сделки с **отрицательной** стадией тоже — в них оплаты **не ищем**. То есть в кандидаты
попадают только сущности из «живых» стадий.

**Что выбираем:**
- **Сумма и валюта обязаны совпадать** (решение владельца, без переключателя): инвойс или оплату
  сделки с суммой или валютой, **отличной** от платежа, **выбрать нельзя** — исключаем из кандидатов;
- найдены **и инвойс, и оплата**, причём **инвойс связан с этой сделкой** (`parentId`) — это одна цель
  в двух представлениях → предпочтение **инвойсу**; **если связи нет** — это **разные** кандидаты;
- **нет ни одного точного совпадения** (частичная/групповая оплата — сумма не сходится) → сущность
  **не трогаем**, кладём в **«очередь ручного разбора»** и уведомляем;
- **одно** совпадение → авторазнесение;
- **несколько разных совпадений** → авторазнесение на **минимальный ID** **и оповещение в чат** (тот
  же чат, куда шлём ошибки), чтобы человек проверил неоднозначность. (Полные дубли одной сущности —
  тот же ID — схлопываются молча, это не неоднозначность.)

**Порядок и ветки поиска (при включённом авторазнесении — а значит, есть смарт-процессы).** Сначала
ищем **инвойс**; **не нашли инвойс → ищем среди оплат сделок**. Итог:
- **найдено** → размечаем по «Что выбираем» выше (одно → разносим; несколько → минимальный ID +
  оповещение в **чат ошибок**);
- **не найдены ни инвойс, ни оплаты** → элемент уже создан (Этап D), сумма остаётся в поле «**осталось
  распределить**», пользователь распределяет **вручную**; проблема-оповещение «**некуда разнести**» идёт
  в **чат ошибок** (§5). Ответственным записи остаётся ответственный за клиента/мою компанию.

**Авторазнесение выключено** → авто-поиск **не запускаем**, ветки «не найдено» не возникают;
распределение — вручную из элемента (§3). Кнопка **«разнести автоматически»** из элемента запускает
**тот же** поиск для этого элемента, но **новый элемент не создаём** (он уже существует).

**«Очередь ручного разбора»** — не новый открытый эндпоинт: это **состояние** элемента смарт-процесса
(поле **«осталось распределить»** > 0 — **стадий у элемента нет**) / дела, под **нативными правами
CRM** Б24, а не отдельная незащищённая выборка.

**Действия:**
- **Сделка** — стадию **напрямую не меняем**. Отметка **«оплачено»** в оплате сделки
  ([crm.item.payment.pay](https://apidocs.bitrix24.ru/api-reference/crm/universal/payment/crm-item-payment-pay.html)
  — меняет **только статус оплаты**, доп. полей не принимает) + **отдельный вызов триггера**
  ([`crm.automation.trigger.execute`](https://apidocs.bitrix24.ru/api-reference/crm/automation/triggers/crm-automation-trigger-execute.html):
  `CODE`+`OWNER_TYPE_ID`+`OWNER_ID`; сумму/валюту/назначение передаём через него, не через `payment.pay`).
- **Инвойс** — стадия по настройке (указана → перевести; не указана → не трогаем) и/или **триггер в инвойсе**.

> **Состояние (мутационный слайс).** За опт-ин гейтом **`autoDistribute`** (в настройках, default OFF)
> **реализованы две amount-цели**: отметка оплаты сделки (`crm.item.payment.pay`, `deal-payment`) и **смена
> стадии инвойса** (`crm.item.update` на `allocation.invoicePaidStageId`; **не указана ⇒ инвойс не трогаем**).
> Чистый билдер `app/utils/allocationMutation.ts` + транспорт `server/utils/allocationMutationWrite.ts`
> (конверт-aware applied-детект: `payment.pay` → `{result:true}`, `crm.item.update` → `{result:{item}}`),
> идемпотентный порядок **mutation-before-fact** (пре-чек `isTargetApplied` читает состояние цели в B24 —
> Фаза A → мутация → write-once факт; уже-применённая цель на редоставке: пропуск оплаты + запись факта для
> учёта/сторно; REST-ошибка бросается ДО факта → чистый ретрай). Счётчик `distributed`. Подтверждено вживую
> (`pnpm mutate:test` + apply/revert стадии инвойса на seed-счёте; state-read Фазы A — на `bel.bitrix24.by`).
>
> **Триггер: билдер + транспорт + настройка готовы, проводка — нет.** Чистый `buildTriggerExecution(target,
> {triggerCode})` (`app/utils/allocationMutation.ts`) строит `crm.automation.trigger.execute` (`CODE`+`OWNER_TYPE_ID`+
> `OWNER_ID` — единственные параметры метода; `OWNER_TYPE_ID`: сделка=2, смарт-процесс=его `entityTypeId`; доп.
> сумму/валюту метод НЕ принимает), транспорт `executeTriggerViaRest` (`server/utils/allocationMutationWrite.ts`)
> шлёт его и читает `{result:true}`; `CODE` хранится в `allocation.triggerCode` (маска `[a-z0-9.\-_]`, fail-safe —
> нет/битый ⇒ не трогаем). Всё покрыто тестами.
> **Проводка триггера в hot-path — сделана (#79, best-effort):** в `crm-sync` за тем же гейтом `autoDistribute`
> + заданный `triggerCode` каждая **распознанная trigger-цель** (сделка/смарт-процесс, дедуп по kind+id) проходит
> `hasAllocationFact` (пре-чек — редоставка не пере-триггерит) → `applyTrigger` → при подтверждённом firing
> write-once факт (`recordAllocation`, счётчики `allocated`+`distributed`). **`applyTrigger` — best-effort, как
> `notifyChat`:** триггер **сигналит** «деньги пришли» (сам денег не двигает), поэтому его сбой **никогда не роняет
> батч** — любая ошибка (транзиентная ИЛИ постоянная: `CODE` задан, но не зарегистрирован → «...is not registered»)
> глотается+логируется, факт не пишется (ни storm'а падений). Транспорт `crm.automation.trigger.execute`
> идёт **по OAuth-резолверу воркера** (контекст приложения есть — вебхуку бы вернулось «Application context required»).
> **SINGLE-SHOT (важно):** триггер пробуется **один раз** — на первой обработке операции с совпавшей компанией. B24-маркер
> дедупа пишется **независимо** от исхода триггера, поэтому повторная доставка/опрос отсекается верхним гейтом и до
> триггер-цикла не доходит. Значит промах первой попытки (транзиентная ошибка, проглоченная best-effort, ИЛИ `CODE` задан,
> но ещё не зарегистрирован) **НЕ пере-пробуется** — firing теряется, а не «само-заживает». Это принятая v1-семантика (`CODE`
> регистрируется на установке, до потока операций); долговременный ретрай триггера — follow-up.
> **Регистрация триггера на установке — сделана и live-verified (#79):** `/install` идемпотентно регистрирует
> канонический триггер приложения `crm.automation.trigger.add` (CODE `cba_payment_received` из `B24_PAYMENT_TRIGGER`,
> чистый билдер `buildTriggerRegisterCall`) — админ портала вешает этот `CODE` на своё правило автоматизации, а в
> настройках указывает его в `allocation.triggerCode`. Best-effort (не-админ/некоммерческий тариф → не блокирует
> установку, статус — в диагностике). Метод требует контекста приложения (OAuth) + прав админа. **Подтверждено
> вживую** тем же билдером через OAuth-транспорт (`pnpm trigger:test --apply --fire` на `bel.bitrix24.by`).
> **Firing — тоже LIVE-VERIFIED (#79):** тот же прогон делает `executeTriggerViaRest`→`crm.automation.trigger.execute`
> `{result:true}` на **сделке** (OWNER_TYPE_ID=2) **и элементе смарт-процесса** (OWNER_TYPE_ID=его `entityTypeId`=1044);
> незарегистрированный CODE → `Trigger ... is not registered` (валидирует best-effort-глоток воркера). Метод требует
> контекста приложения (OAuth) — вебхуком вернулось бы «Application context required». Реакция правила автоматизации на
> CODE — за админом портала (наш код доставляет сигнал `{result:true}`, это и есть его задача).
> **Осталось (#79):** (1) путь заказа `payment.add`; (2) **долговременный ретрай триггера** (сейчас single-shot:
> промах первой попытки не пере-пробуется — B24-маркер дедупа пишется независимо от исхода). До регистрации `CODE`
> проводка молча не срабатывает (best-effort).

**Суть триггера:** сигнал «**пришли деньги**» (по возможности — с суммой и валютой). Решение «вся сумма
или часть», «сюда или в соседний инвойс» **клиент принимает сам** через свои бизнес-процессы (БП). Если
REST-триггер позволяет передать доп. параметры (сумма/валюта/назначение) — **передаём их**
(см. [документацию триггеров](https://apidocs.bitrix24.ru/api-reference/crm/automation/triggers/index.html)).

Смарт-процесс — пользовательский объект:
[user-defined object types](https://apidocs.bitrix24.ru/api-reference/crm/universal/user-defined-object-types/index.html).
Привязка элемента к двум компаниям (моя + клиент) — **отдельными полями элемента**, не нативной
мульти-привязкой. **У элемента стадий НЕТ** — вместо стадии есть **read-only поле «осталось
распределить»** (сколько денег ещё не разнесено; редактировать нельзя, считает приложение). **Переход тарифа**: кнопка «Создать смарт-процесс» при апгрейде; если тариф не
поддерживает — уведомление, что смарт-процесс недоступен. **Апгрейд** (появились смарт-процессы):
уже созданные **дела остаются как есть** (ничего не мигрируем — по аналогии: при **даунгрейде** мы не
удаляем накопленные элементы смарт-процесса), **новые** операции идут в смарт-процесс.

### Этап E. Оповещение (в обеих ветках)
1. **Дело** — конфигурируемое дело в таймлайне, с кнопками (§6). В ветке без смарт-процессов это
   и есть основная запись об операции.
   [configurable timeline activity](https://apidocs.bitrix24.ru/api-reference/crm/timeline/activities/configurable/structure/layout.html)
2. **Сообщение чат-бота** в выбранный чат (в настройке: **какой чат** + **что писать** —
   приходы / расходы / всё). Тексты (сумма — с кодом валюты):
   - **оплата по сделке**: дата · контрагент · `сумма BYN` · основание · **ссылка на сделку** ·
     ответственный за сделку;
   - **инвойс**: дата · контрагент · `сумма BYN` · основание · **ссылка на инвойс** ·
     ответственный за инвойс.

## 3. Распределение денег (в смарт-процессе)

Режим задаёт настройка **«распределять автоматически»** (доступна **только при поддержке
смарт-процессов**):
- **включена** → приложение разносит деньги **автоматически**; ручной разнос **заблокирован**
  (кнопки ручного разноса показывают сообщение, что включён авто-режим);
- **выключена** → разносим **вручную** из элемента; дополнительно есть кнопка **`разнести автоматически`**
  — разово запускает то же действие, что и авто-режим, для этого элемента.

Эта же настройка **включает/выключает работу триггера**: включена → триггер срабатывает (при разносе);
выключена → автоматически триггер **не** срабатывает (только при ручном разносе).

Кастомные поля-**кнопки** в элементе смарт-процесса (мы их создаём):

- **`повторно поискать клиента`** — деньги пришли, а клиент не найден: пользователь вручную вносит
  **расчётный счёт** (в карточку клиента / реквизиты), затем жмёт кнопку в элементе → приложение
  **заново ищет компанию** по счёту, находит и (при возможности) распределяет.
- **`поискать что оплачено`** — распределить деньги по инвойсам/оплатам вручную (при **выключенном**
  авто-разносе):
  1. берём **элемент** (операцию) и его **сумму**;
  2. ищем **инвойсы/оплаты** для покрытия — **в рамках МОЕЙ компании и компании-клиента**
     (при нескольких юрлицах иначе будет путаница);
  3. **кладём деньги**: оплата → «оплачено», инвойс → оплаченная стадия.
- **`разнести автоматически`** — доступна при **выключенной** галке; разово запускает авто-разнос
  для этого элемента.

Если клиент или моя компания не определены — по нажатию кнопки пользователь сразу увидит проблему.

### Реакция на изменения после распределения
- **Авто-распределение** — последующая **смена стадии инвойса** или **снятие оплаты** **никак не влияет**
  на элемент смарт-процесса (повторно ничего не делаем).
- **Ручное распределение** — **смена стадии инвойса** или **снятие оплаты** → элемент **требуется
  распределить снова** (помечаем как «требует распределения», пользователь разносит заново).

## 4. Параметры, которые учитывает приложение

- **Распознавание из назначения платежа — по МАТРИЦАМ (маскам), а не «слово + число».** В настройках —
  **несколько матриц**; каждая описывает формат идентификатора шаблоном + человекочитаемым разъяснением
  и привязана к **типу идентификатора** (→ полю сущности, таблица ниже). Идентификатор — **буквы, цифры
  и разделители** (`-`, `/`, `.`, `№`, …) **без пробела**.
  - Шаблон: `d` = цифра, литеральные символы (буквы/дефис/слэш) — как есть. Примеры:
    - `dddd` — «четыре цифры» (напр. `2001`);
    - `ВОРС-dddd` — «ВОРС-дефис-4 цифры» (напр. `ВОРС-1234`);
    - `BOPC-ddd/dd` — «BOPC-3 цифры-слэш-2 цифры» (напр. `BOPC-123/45`).
  - **Матрицы без литерального якоря** (`dddd` в чистом виде) — слабые: совпадут с датой/суммой в тексте.
    Для них желателен литеральный префикс (`СЧ-dddd`) **или** скоуп по полю, иначе высок риск ложных.
  - **Нормализация алфавита (гомоглифы).** Настройка «приводить к **кириллице** или **латинице**»:
    `ВОРС` (кир.) и `BOPC` (лат.) визуально идентичны (`В↔B О↔O Р↔P С↔C Е↔E А↔A К↔K М↔M Н↔H Т↔T Х↔X У↔Y`).
    Перед сравнением и назначение, и матрицу приводим к выбранному алфавиту, иначе номер не совпадёт.

  **Поля, из которых берётся номер** (тип идентификатора → поле сущности Б24):

  | Тип (`IdentifierKind`) | Поле сущности |
  |---|---|
  | `invoice-number` | «Номер» счёта-инвойса (Smart Invoice `accountNumber`) |
  | `invoice-id` | `id` инвойса |
  | `deal-id` | `id` сделки |
  | `deal-field` | настраиваемое поле сделки (`UF_CRM_*`) — **на каждое направление сделок** |
  | `order-id` | `id` заказа |
  | `order-number` | `accountNumber` заказа |
  | `payment-id` | `id` оплаты |
  | `payment-number` | `accountNumber` оплаты |
  | `smart-id` | `id` элемента смарт-процесса |
  | `smart-field` | настраиваемое поле элемента (`UF_*`) — **на каждый смарт-процесс и направление** |
  | `document-number` | `number` документа «Генерации документов» (мост → `entityTypeId`/`entityId`) |

  **Как настроить распознавание (пока — через `app.option`, UI-формы ещё нет).** Настройки лежат в
  `app.option` под ключом `cb_settings_v1`, блок `recognition`:
  - `matrices` — массив `{ mask, kind }` (маска: `d` = цифра, прочее — литерал; `kind` — из таблицы выше).
    Без матриц распознавание не срабатывает **ни для чего**.
  - `alphabet` — `cyrillic` | `latin` (сведение гомоглифов `ВОРС`↔`BOPC`).
  - `configFields` — параметры для config-driven кайндов (ключ → значение):
    - `"deal-field"` → имя поля сделки (`UF_CRM_*`), где лежит номер (для `deal-field`);
    - `"smart-entity"` → **`entityTypeId` смарт-процесса строкой** (напр. `"1032"`; узнать — `crm.item.fields`/список СП),
      нужен для `smart-id` и `smart-field`; невалидное/пустое ⇒ кайнд `unsupported` (молча, без записи);
    - `"smart-field"` → имя поля элемента СП (`UF_*`), где лежит номер (для `smart-field`, вместе с `smart-entity`).

  Пример: `{"recognition":{"alphabet":"cyrillic","matrices":[{"mask":"СЧ-dddd","kind":"invoice-number"},{"mask":"ЗАК-dddd","kind":"deal-field"}],"configFields":{"deal-field":"UF_CRM_1699999999"}}}`.

  Точные REST-имена полей фиксируем в [`REST_METHODS.md`](REST_METHODS.md) на REST-слайсе. Один
  идентификатор может вести к нескольким сущностям — неоднозначность разрешается по §2 (сумма+валюта,
  минимальный ID + оповещение). **Уточнение по счетам:** номер счёта (`accountNumber` смарт-счёта)
  **уникален в пределах портала** (Б24 не даёт создать два счёта с одним номером), поэтому
  `invoiceLookup(номер+компания)` находит **не более одного** счёта — неоднозначность нескольких
  кандидатов возможна только **между разными видами целей** (счёт vs оплата сделки vs …), а не среди
  самих счетов.

  > **Подтверждено на живом портале (2026-07-06).** Смарт-счёт (`crm.item.fields entityTypeId=31`):
  > `accountNumber` (номер), `companyId` (клиент), `mycompanyId` (наша компания), `stageId`,
  > `opportunity` (сумма, double), `currencyId`. Поиск — `crm.item.list` (ответ `{result:{items:[…]}}`).
  > Отрицательная стадия — по справочнику `crm.status.list`, `SEMANTICS='F'` (напр. «Не оплачен»
  > `DT31_11:D`). Реализовано в `server/utils/invoiceLookup.ts` (первый REST-слайс #109).
  >
  > **«Моя компания» (Этап C)** — не отдельная сущность, а **компания** (`entityTypeId=4`) с флагом
  > `isMyCompany='Y'`; `mycompanyId`/`companyId` счёта — ссылки типа `crm_company`. Значит поиск моей
  > компании по нашему счёту идёт тем же путём, что и клиента (`companyLookup` по `RQ_ACC_NUM`), только
  > результат дополнительно должен нести `isMyCompany='Y'`.
  >
  > **Расчётный счёт (`RQ_ACC_NUM`) НЕ уникален** (подтверждено вживую): один и тот же счёт можно завести
  > на **несколько компаний** (и даже несколько банк-деталей одного реквизита). Значит `companyLookup` по
  > счёту может вернуть **несколько** компаний — берём первую (принятый дефолт). (Не путать с номером
  > **счёта-фактуры** `accountNumber` — тот уникален, см. выше.)
  >
  > **Оплата сделки (`deal-payment`) — это объект `crm.item.payment`**, не привязанный счёт. Создаётся:
  > товарная позиция сделки (`crm.item.productrow.add`) → `crm.item.payment.add` → `crm.item.payment.product.add`
  > (задаёт сумму) → `crm.item.payment.pay`. **Отмена/удаление оплаченной оплаты требует scope `sale`**
  > (`crm`-only токен → `insufficient_scope`; удалить сделку с активной оплатой тоже нельзя — «У заказа
  > есть активные оплаты»). Учесть при проектировании #109 (проводка оплаты) и при правах приложения.
  >
  > **`RQ_IIK` (белорусский ИИК)** отвергает 28-символьный IBAN (ошибка «Array», даже с валидной
  > контрольной суммой и белорусским шаблоном реквизитов; значения ≤20 символов проходят). Поэтому счёт
  > кладём в свободный `RQ_ACC_NUM`; `companyLookup` его же ищет первым.
  >
  > ⚠ **Грабли удаления (для будущего prod-cleanup — `ONAPPUNINSTALL`, ручной разбор ошибок):** реквизиты
  > и банк-детали **не** каскадируются при удалении компании, и если удалить компанию **раньше** её
  > реквизитов — Б24 теряет права на осиротевшие реквизиты (`crm.requisite.delete`/`bankdetail.delete` →
  > «Access denied»), а «зомби»-банк-деталь навсегда остаётся в поиске по `RQ_ACC_NUM`. Порядок: сначала
  > банк-деталь → реквизит → компания (пока компания жива, доступ есть). Проверено на живом портале seed-скриптом.
- **Направление** — авто-проведение работает **и для приходов, и для расходов** (обе стороны).
- **Распределять автоматически** — вкл/выкл (только при смарт-процессах). Вкл → авто-разнос, ручной
  заблокирован; выкл → ручной разнос + кнопка «разнести автоматически» (§3). Также **вкл/выкл работу
  триггера** (§2 Этап D).
- **Стадия инвойса при оплате** — указать стадию (перевести в неё) или **не указывать** (стадию не
  трогаем); смена также возможна **триггером в инвойсе**. Для **сделки** стадию не задаём — только
  отметка «оплачено» + триггер.
- **Уведомления** — какой чат; что писать (приходы / расходы / всё).
- **Чат ошибок** — отдельный **чат** (из настроек), куда приложение шлёт ошибки/проблемы обработки (§5).
  Именно чат, а не конкретный пользователь: **универсальнее** — переживает отпуск/увольнение сотрудника,
  виден всей команде, не зависит от одного адресата.
- **Фильтры-исключения** — список **расчётных счетов** (опц. названий) контрагентов: не загружать /
  не уведомлять.
- **Хранение** — смарт-процесс (если тариф поддерживает) либо список дел; кнопка перехода при апгрейде.
- **Кнопки в деле** — набор действий (§6).

## 5. Ошибки, UNMATCHED и ручная корректировка

Ошибки и проблемы обработки приложение шлёт в указанный в настройках **чат ошибок** (§4).

> **Формат сообщений об ошибках.** Каждое сообщение **явно указывает, какое приложение рапортует**
> (напр. префикс «[Импорт выписки из клиент-банка]») — чтобы в общем чате было ясно, кто пишет. **Тон —
> деловой**: коротко и по сути (что за операция, что не так, что делать), без эмодзи-«воды» и фамильярности.
> Это же правило тона распространяется и на обычные оповещения (§2 Этап E).

- **Ошибка файла/данных** — оповестить в чат ошибок; при необходимости — **создать issue**.
  ⚠️ **Банковская тайна.** Файл выписки (счета, суммы, контрагенты) в **публичный** issue по умолчанию
  **не прикладываем**. Варианты по возрастанию: (1) маскированный фрагмент (счета/суммы частично
  скрыты — хватает для диагностики парсера); (2) приватный канал (приватный трекер / почта с
  ограниченным доступом) для полного файла; (3) публичный issue с файлом — **только по явному
  активному подтверждению** («да, понимаю, файл станет публичным»). Канон «без PII по умолчанию» —
  как в [`REFACTOR_PLAN.md`](REFACTOR_PLAN.md) (#61).
- **UNMATCHED — клиент не определён** (компания-контрагент не найдена по счёту) — запись всё равно
  создаётся, но привязывается к **моей компании** (из §2 Этап C.1): на тарифе со смарт-процессами —
  **элемент** со связью только с моей компанией, без него — **дело** в таймлайне моей компании.
  Ответственный записи — **за мою компанию**; проблема-оповещение идёт в **чат ошибок**, чтобы её
  точно заметили.
- **Моя компания не определена** (не нашли по нашему счёту — §2 Этап C.1) — привязать запись **не к
  чему**: **в CRM ничего не пишем**, шлём проблему **в чат ошибок** с причиной («наш счёт … не
  сопоставлен ни с одной моей компанией»; если и клиент не найден — указываем обе причины); оператор
  заводит реквизит и повторяет обработку. Это — один из **двух** случаев, когда запись в CRM **не
  создаётся** (второй — **исключение**, Этап A2). Отдельно стоит **невалидная операция** (Этап A1) — она
  не записывается как обычная, а рапортуется ошибкой (выше).
- **Структура смарт-процесса повреждена/удалена** (админ удалил элемент или поле, которое должно
  быть): ловим «не хватает того, что должно быть» → пишем проблему **в чат ошибок**, **пересоздаём
  структуру** и **повторяем** обработку операции. Если со **2-й** попытки снова ошибка — пишем проблему
  **в чат ошибок** и дальше **не зацикливаемся**.
- **Ручная корректировка** — на **деле** или в **смарт-процессе**. При изменении привязок
  **ничего автоматически не пересчитывается** — пересчёт запускает **сам пользователь** кнопками
  элемента (§3: `повторно поискать клиента` / `поискать что оплачено` / `разнести автоматически`).

## 8. Открытые вопросы (к стадии реализации)

Не решаются этим документом — фиксируются как кандидаты в issue перед реализацией:

- ✅ **Распознавание по матрицам (маскам)** (§4) — `app/utils/purposeMatch.ts` **реализован**
  (`recognizeByMatrices` + `foldHomoglyphs`, матрицы-шаблоны + нормализация алфавита). Остаётся:
  **калибровка матриц** на 5–10 реальных строк назначения (фикстуры) и **хранение матриц/алфавита в
  настройках портала**. Открытые вопросы к REST-слайсу: (а) `value` = весь совпавший фрагмент с
  префиксом (`СЧ-1234`) — подтвердить, ищем по нему целиком или по «полезной» части; (б) гибкость
  разделителей (несколько масок-вариантов на один человеческий формат).
- **Карта сопоставления «тип идентификатора → поле сущности»** (§4, таблица полей) — модель настройки:
  на каждое направление сделок и на каждый смарт-процесс/направление указать, какое поле держит номер
  (точные REST-имена полей — в `REST_METHODS.md`). Плюс мост через `crm.documentgenerator.document.list`.
- **Обнаружение изменений** (§3: смена стадии инвойса / снятие оплаты → «требует распределения») —
  транспорт события: вебхук `ONCRMDYNAMICITEMUPDATE` / подписка на поле / поллинг.
- **Определение тарифа** (поддержка смарт-процессов) — метод REST (напр. проба `crm.type.list`).
- **Несколько «моих» компаний на один счёт** — `findMyCompanyByAccount` сейчас берёт **первую** молча
  (как и для контрагента: `RQ_ACC_NUM` не уникален). Но для **моей** компании цена ошибки выше (она
  определяет ответственного и куда пишется дело). Решить: такой кейс невозможен на практике (один наш
  счёт → одна наша компания) или при неоднозначности эскалировать в **чат ошибок** (§5), а не брать первую.
- **Кэш «нашего счёта»** — `findMyCompanyByAccount` добавляет до 3 REST-вызовов на операцию, а наш счёт
  почти постоянен в пределах портала/батча. На этапе проводки в `crm-sync` кэшировать этот поиск (по счёту),
  чтобы не жечь квоту (classic REST ~2 rps) на каждую операцию пачки.
- **Права приложения на оплаты сделок (scope `sale`)** — сама **проводка** оплаты сделки
  (`crm.item.payment.pay`) идёт под scope **`crm`** и уже реализована (за гейтом `autoDistribute`). Scope
  **`sale`** **добавлен** в `B24_REQUIRED_SCOPES` (`crm`, `sale`, `im`, `user_brief`, `placement`, #172) — он
  нужен для **`order-id`** (`sale.payment.list` по `orderId` → id оплат заказа, которые пересекаются с
  company-пулом; `crm.item.payment.list` `orderId` не отдаёт). ⚠ Добавление `sale` **потребует ре-consent**
  на уже установленных порталах. Тот же scope покроет будущее **сторно** оплаты (`sale.payment.update PAID=N`,
  снятие блокировки «у заказа есть активные оплаты»; в проде путь отмены разнесения ещё не реализован — в dev
  только `mutate:test --revert`). Целевая модель факта «разнесено/откат» — состояние объекта в B24; **сейчас**
  факт хранится в `allocation_fact`, его снятие — Фаза A (§1).
- **`unp` пусто vs undefined** — тип `StatementParty.unp: string` обязателен; зафиксировать конвенцию
  (пустая строка = «нет УНП»), чтобы фильтры/отображение не разошлись.
- **Rate-limit REST** — на операцию теперь 2 поиска компании + поиск инвойсов/оплат; учесть рост
  вызовов до реального объёма (#191 — дизайн/остаток в `docs/QUEUES.md` «REST-бюджет проводки платежей»;
  связать с #90/#91).
- ✅ **Критерий совпадения и поведение при неоднозначности — решены:** кандидат = совпали сумма и
  валюта (строго, без переключателя); из кандидатов исключены инвойсы/сделки с отрицательной стадией.
  Нет точного совпадения → ручной разбор; одно → авторазнесение; несколько → минимальный ID **+
  оповещение в чат**. См. врезку в преамбуле и §2.
