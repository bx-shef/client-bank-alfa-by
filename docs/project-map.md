# Карта проекта — client-bank-alfa-by

> Last reviewed: 2026-07-01

Канонический срез состояния проекта: **цель, шаги, что сделано / сейчас / дальше / потом,
что мешает запуску и что после**. Источник правды для навыков `/report-status`,
`/report-questions` (reporting-kit). Держим синхронно с [`REFACTOR_PLAN.md`](REFACTOR_PLAN.md)
(детальная дорожная карта) и [`CLAUDE.md`](../CLAUDE.md) (архитектура/файлы).

## Цель

Приложение Bitrix24, которое **автоматически переносит банковскую выписку в CRM**:
- забирает операции по счетам компании — **онлайн** из Альфа-Банка Беларусь (OAuth API) и
  Приорбанка (Open Banking СПР), либо **ручной загрузкой** стандартного файла выписки;
- **находит компанию-контрагента** в CRM по расчётному счёту/УНП;
- пишет каждый **приход/расход как дело** в таймлайне (дедуп, без дублей);
- **оповещает в чат** по настраиваемым правилам (какой чат, приходы/расходы, исключения).

Публичная часть — лендинг (SSG). Внутри портала — iframe-страницы (просмотр выписки, настройки,
установка). Серверная часть (Nitro) — приём событий портала, хранилище токенов, дальше опрос банков
и запись в CRM.

## Шаги (дорожная карта, «от малого к сложному»)

1. Доменное ядро + первый UI-срез на mock-данных.
2. Встройка в Bitrix24 (dual-mode SDK, `/install`, iframe-страницы).
3. Backend PoC: OAuth Альфы + приём событий портала Б24 (install/uninstall).
4. Поиск компании по корр-счёту + запись универсального дела.
5. Cron-опрос банков (разделение приход/расход, фильтры).
6. Сообщения в чат по правилам.
7. Docker + деплой (фронтенд и backend).
8. MCP-сервер по выписке.
9. Второй/третий провайдер: Приорбанк (Open Banking) + ручная загрузка файла.

Подробные критерии приёмки по каждому шагу — [`REFACTOR_PLAN.md`](REFACTOR_PLAN.md).

## Что сделано

- **Доменное ядро** (чистое, покрыто тестами, переносимо в backend): модель выписки и **единый
  интерфейс** `StatementNormalizer` (`raw → StatementItem[]`); классификация приход/расход, дедуп
  `account|docId`, фильтр чата; билдер универсального дела.
- **Нормализаторы всех трёх провайдеров** дают один выход: Альфа (`normalizeAlfa`), Приорбанк
  (`normalizePrior`, подтверждён на живом sandbox), ручная загрузка (`normalizeClientBank` — текст
  клиент-банка `***** ^Type=`, покрыт на образцах BYN/CNY).
- **OAuth-ядра** банков (чистые, без транспорта/подписи): Альфа (`alfaOauth.ts`), Приорбанк
  (`priorOauth.ts` — OAuth/DCR/consent). Live-скрипты разведки для обоих банков (единая обвязка).
- **Встройка в Bitrix24**: `useB24` (dual-mode), страницы `/app`, `/settings`, `/install`; лендинг `/`.
- **Backend (Nitro, слайс)**: эндпоинт вебхуков `POST /api/b24/events` (install/uninstall,
  fail-closed вердикт `application_token`), хранилище токенов портала в Postgres (refresh шифруется
  AES-256-GCM), настройка уровня приложения через `app.option` по фрейм-токену, health-эндпоинт
  `GET /api/health`.
- **Шина очередей (BullMQ + Redis), фундамент (Фаза 1)**: контракты очередей `server/queue/topology.ts`
  (имена очередей `b24-events`/`bank-fetch`/`file-parse`, payload'ы, идемпотентные `jobId`) +
  `server/queue/connection.ts` (ленивый `getQueue`, гуард `REDIS_URL`); Redis в изолированной сети
  `queuenet`. Продюсеры и воркер-контейнер — далее (Фаза 2).
- **Деплой**: фронтенд и backend собираются в Docker-образы (GHCR + Watchtower за nginx-proxy),
  один домен, nginx проксирует `/api/*` в backend; CI (lint/test/typecheck/generate + сборка образов).

## Что сейчас

- Ручная загрузка (`manual`): ядро разбора+нормализации готово для **двух форматов** — `***** ^Type=`
  (Приор/Альфа-выгрузка) и `1CClientBankExchange` (обмен 1С, РФ+РБ), диспетчер `manualImport.ts`,
  покрыто тестами; следующий шаг — UI-загрузка файла и включение провайдера (`implemented:true`).
- Приорбанк: подтверждён на sandbox; следующий шаг — серверный движок опроса поверх `priorOauth.ts`.

## Что дальше

- **UI ручной загрузки файла** + проверка нормализатора на реальных выписках (особенно инвалютных).
- **Серверный движок опроса** (Альфа/Приор) как воркеры очередей (`bank-fetch`, follow-up `b24-events`):
  HTTP-транспорт поверх готовых OAuth-ядер, refresh-цикл, запись операций (Фаза 2 шины).
- **Поиск компании по корр-счёту** + запись универсального дела (todo-API) с дедупом на backend.

## Что потом

- Cron-опрос с фильтрами (по р/счёту и теме платежа), сообщения в чат по правилам.
- Хранение настроек на реальных данных (`app.option`) вместо demo-localStorage.
- Сопоставление платежей с заказами/инвойсами 1:1; сохранение в смарт-процесс (если тариф поддерживает).
- MCP-сервер по выписке. Поддержка ещё стандартных форматов (camt.053, MT940, CSV — 1С уже есть).

## Что мешает запуску

- **Живые прогоны банков — только из Беларуси**: sandbox/прод Альфы и Приора недоступны из облака
  (гео-блок) — OAuth и выписку прогоняем с BY-сервера.
- **Приорбанк прод — СКЗИ (BY-crypto TLS)**: нужен крипто-шлюз АВЕСТ + сертификат ГосСУОК (issue #41);
  на sandbox не требуется.
- **Способ auth токен-эндпоинта Альфы** (Basic vs form-body) — сверить до боевого backend (issue #26).
- **Ручная загрузка**: до включения в UI — внутренний лимит размера в парсере и проверка инвалютного
  правила суммы на реальных выписках (issue #19).
- **Точная встройка в портал** (`placement.bind`/`event.bind`, `NUXT_PUBLIC_SITE_URL`) — добить на
  тестовом портале.

## Что после запуска

- Мониторинг (health-эндпоинт уже есть), ротация секретов, наблюдение за rate-limit опроса.
- Расширение списка банков/форматов по мере запросов клиентов.
- Гигиена репозитория при появлении команды (PR-шаблон, CODEOWNERS — issue #5).
