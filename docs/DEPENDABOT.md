# Dependabot — как работать

> Last reviewed: 2026-06-29

Как у нас устроены автообновления зависимостей и что с ними делать день в день.
Извлечено из базы знаний `ai-agent` (`docs/08_git-ci-di/03_repo_protection_ci.md`,
`02_owner_setup.md`, `05_security_checklist.md`) и адаптировано под наш
[`.github/dependabot.yml`](../.github/dependabot.yml).

## Две части настройки

Dependabot включается в **двух местах**, и обе нужны:

1. **Тумблеры в Settings репозитория** (делает владелец, admin) — см.
   [`REPO_SETUP_CHECKLIST.md` → Dependabot](./REPO_SETUP_CHECKLIST.md#3-dependabot).
   - **Dependabot alerts** — уведомления о новых CVE в зависимостях.
   - **Dependabot security updates** — авто-PR на патч уязвимой версии.
   - **Dependabot version updates** — плановые обновления по расписанию
     (работают только при наличии файла `dependabot.yml`, он уже в репозитории).
   - Settings → Actions → General → ☑ **Allow GitHub Actions to create and approve
     pull requests** — иначе Dependabot не сможет открывать PR.
2. **Файл `.github/dependabot.yml`** (в репозитории) — описывает, *что* и *как часто*
   обновлять. Уже на месте; разбор ниже.

> **Renovate мы не используем** — версии ведёт Dependabot. (Если бы использовали
> Renovate, version updates Dependabot отключали бы, чтобы два бота не плодили
> дублирующие PR; alerts + security updates оставляли бы включёнными.)

## Наш `dependabot.yml` — что он делает

| Экосистема | За чем следит | Расписание |
|---|---|---|
| `npm` | `package.json` / `pnpm-lock.yaml` в корне | weekly, понедельник; лимит 5 PR |
| `github-actions` | версии actions в `.github/workflows/*.yml` | weekly, понедельник; один групповой PR |

**Группировка** (главное средство против шума — связанные пакеты в один PR):

- `nuxt` — `nuxt` + `@nuxt/*` идут в ногу одной версией.
- `b24` — весь вендорный scope `@bitrix24/*` одной пачкой.
- `dev-deps` — все dev-зависимости одним PR.
- `actions` — паттерн `"*"`: один PR на все бампы экшенов (forward-proof: новый
  экшен попадёт в ту же группу).

> Runtime-зависимости вне именованных групп (`@vueuse/*`, `tailwindcss`) приходят
> отдельным PR на каждую — для маленького проекта это нормально. Если их станет
> много и PR-шум вырастет — добавить общую группу `production` или
> `minor-and-patch` (см. справочник в базе знаний).

**Docker — готовый шаблон, пока закомментирован.** В `dependabot.yml` уже лежит
блок `package-ecosystem: docker` (группа `"*"` + осознанный `ignore` major `node`
с обоснованием, как в эталоне `currency-converter`), но закомментированный: без
`Dockerfile` экосистеме docker нечего сканировать. Когда появятся деплой-артефакты —
**раскомментировать блок**, других правок не нужно.

## Как обрабатывать PR от Dependabot

Это **не отдельный процесс — тот же гейт, что у обычного PR**:

- Каждый Dependabot-PR прогоняется через наш `ci`-workflow (lint → test →
  typecheck → generate). Красный CI = merge заблокирован тем же ruleset `protect-main`.
- Секреты на `pull_request` для Dependabot-PR недоступны, `GITHUB_TOKEN` read-only —
  это нормально и безопасно (джоба, которой нужен секрет, на таком PR не запустится).
- Порядок: посмотреть changelog → дождаться зелёного CI → смержить. **Major-бампы —
  всегда руками с ревью** (могут ломать API).

## Игноры — только осознанно и с комментарием

Если major какой-то зависимости заведомо ломает сборку — заигнорить его
**намеренно, с комментарием-обоснованием прямо в `dependabot.yml`** (что делать
вместо автообновления), а не терпеть красный CI на каждом PR. Minor/patch той же
зависимости продолжают приходить. Канонический пример (для будущего docker-блока):
Node 25+ убрал corepack из официальных образов → major `node` игнорируем, major-переход
делаем вручную с фиксом corepack.

## Авто-мерж (опционально, пока не включён)

По умолчанию — ручной merge. При желании можно включить авто-мерж **только patch/minor**
отдельным workflow; **major — всегда руками**. Если будем включать — по правилам
безопасности из базы знаний:

- Триггер **`pull_request`**, не `pull_request_target` (иначе утекают права/секреты).
- `permissions: contents/pull-requests: write` — на уровне **job**, не workflow.
- Фильтр по `update-type` через `dependabot/fetch-metadata` (пин на SHA).
- Критичные runtime-зависимости (крипто, аутентификация, парсеры) — **исключить**
  из авто-мержа даже на patch: скомпрометированный patch-релиз (классика — `xz`)
  пройдёт автомержем как честный багфикс.

## Безопасность: SHA-pinning экшенов

Теги (`@v4`) мутабельны — владелец action может перезаписать тег и подменить код
(supply-chain RCE). Для прод-репозиториев сторонние actions пинят на **полный commit
SHA** с комментарием версии:

```yaml
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
```

Экосистема `github-actions` в Dependabot сама обновляет эти SHA и открывает PR —
ручной труд не растёт. (В текущем `ci.yml` actions пока на тегах для читаемости;
при ужесточении — переводим на SHA, дальше их ведёт Dependabot.)

## Чек-лист при изменении `dependabot.yml`

- [ ] Перечислены все применимые экосистемы (`npm` + `github-actions`; `docker` — когда появится Dockerfile).
- [ ] Единое weekly-расписание; `open-pull-requests-limit` выставлен для `npm`.
- [ ] Связанные пакеты сгруппированы; dev-зависимости — отдельной группой.
- [ ] `github-actions` (и `docker`) — группа с паттерном `"*"`.
- [ ] Заведомо ломающие major'ы — в `ignore` с комментарием-обоснованием.
