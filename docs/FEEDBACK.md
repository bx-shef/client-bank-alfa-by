# Обратная связь: сбор и разбор

> Last reviewed: 2026-07-15

Как в приложении импорта клиент-банка собирается обратная связь, куда она попадает и
как превращается в задачи. Этот документ — **дизайн канала** (что и куда); разбор
собранного — отдельная роль-агент, [`FEEDBACK_TRIAGE_AGENT.md`](FEEDBACK_TRIAGE_AGENT.md).

> **Статус:** дизайн. Сбор отзывов (UI-кнопки, серверный эндпоинт, автозавод issue)
> ещё **не реализован** — этот файл описывает целевую схему; реализация — отдельные
> issue бэклога (см. § «Что делаем дальше»).

## TL;DR

- Два канала отзывов: **сотрудник** (кнопки 👍/👎 в `/app`) и **программа** (воркер
  `crm-sync`, когда «запуталась»: не нашла компанию, неоднозначное разнесение и т.п.).
- Каждый отзыв автоматически становится **issue в приватном репо** (ENV
  `GITHUB_FEEDBACK_REPO`; имя ещё не выбрано — плейсхолдер). Приватность нужна, чтобы
  безопасно хранить данные клиента **и приложенный файл выписки**.
- Разбирает поток ИИ-агент по [`FEEDBACK_TRIAGE_AGENT.md`](FEEDBACK_TRIAGE_AGENT.md):
  группирует, заводит инженерные issue в `bx-shef/client-bank-alfa-by`, закрывает
  разобранное со связкой.

## Зачем отдельный приватный репо

Отзывы почти всегда несут **данные клиента**: номер счёта, суммы, УНП, названия
контрагентов, member_id портала, содержимое выписки. Такое нельзя класть в публичный
репо задач. Поэтому:

- **сырые отзывы + файлы выписок** → приватный репо-приёмник (`GITHUB_FEEDBACK_REPO`);
- **обезличенные инженерные задачи** → репо проекта `bx-shef/client-bank-alfa-by`
  (со ссылкой на приватный отзыв, где лежит конкретика).

Границу держит privacy-guard в [`FEEDBACK_TRIAGE_AGENT.md`](FEEDBACK_TRIAGE_AGENT.md)
(§5.1): агент не переносит контекст/файл из приватного отзыва в публичный issue.

## Каналы

### 1. Сотрудник (`user-feedback`)

Где: страница `/app` (полоса статуса импорта / список операций). Оператор ставит
оценку 👍/👎 и (для 👎/💡) пишет короткий комментарий: «что не так».

- Метка issue: `user-feedback` + `feedback:{positive|problem|suggestion}`.
- `positive` (👍) — сигнал «работает как надо»; в трекинг обычно не заводится.
- `problem` (👎) / `suggestion` (💡) — кандидат на инженерный issue.

> 🎛️ Косметика (из практики кита): голое 👎 демотивирует. Рассмотреть нейтральную
> схему — напр. кнопка **«🔎 Разобрать»** вместо «👎 Проблема». Решение — продуктовое,
> завести отдельным issue.

### 2. Программа — воркер `crm-sync` (`agent-feedback`)

Приложение само сигналит, **когда «запуталось»**. Нужные сигналы уже считаются в
`crm-sync` (см. `CLAUDE.md` → счётчики `unmatched/ambiguous/manual`, `notifyError`):

| Сигнал | Что значит | `feedback:kind` |
|---|---|---|
| `unmatched` | компания по счёту не найдена (нет реквизита `RQ_ACC_NUM`/`RQ_IIK`) | problem |
| `ambiguous` | несколько кандидатов одной суммы — разнесение неоднозначно | problem |
| `manual` | нет точного совпадения — ушло в ручную очередь | problem |
| формат выписки не распознан | падает разбор `Type=X` / 1С-формат | problem |
| fail-open по стадиям | `crm.status.list` вернул пусто для известной категории | problem |

- Метка issue: `agent-feedback` + `feedback:problem`.
- **Серверный дедуп** по корню (напр. `member_id|kind|account`) + **кап N/час**, чтобы
  один сломанный формат не завёл сотни issue (как в оригинальном ките — кап 10/час).

## Формат issue-отзыва (в приватном репо)

```markdown
## Контекст
- member_id: <portal member_id>
- Канал: user-feedback | agent-feedback
- Файл выписки: <имя файла> (вложение ниже)
- Счёт/сделка: <№>, сумма: <...>, версия сборки: <sha>

## Суть
<комментарий сотрудника ИЛИ что распознала/сделала программа>

## Лог обработки
<для agent-feedback: что распознал воркер, какой счётчик сработал, стадия>
```

### Вложение файла

**Ключевое требование.** К отзыву прикладывается **сам файл выписки** (или релевантные
операции), из-за которого возникла проблема — иначе её **не воспроизвести**. Это
безопасно, потому что репо-приёмник приватный.

- Файл кладётся как **вложение к issue** (или в отдельный commit приватного репо со
  ссылкой из тела) — конкретный механизм выбираем на этапе реализации.
- Для `agent-feedback` файл выписки у воркера уже на руках (он его и разбирал) — берём
  оттуда; для `user-feedback` — тот файл, что оператор загружал/смотрел в `/app`.
- Privacy-guard: файл **никогда** не уезжает в публичный репо задач при триаже.

## Поток целиком

```
Сотрудник 👍/👎  ─┐
                  ├─► приватный репо-приёмник (issue + файл выписки)
Воркер crm-sync ─┘        │  метки user-feedback / agent-feedback + feedback:kind
   («запутался»)          │
                          ▼
        ИИ-агент триажа (FEEDBACK_TRIAGE_AGENT.md)
                          │  группирует по корню, сверяет с бэклогом
                          ▼
   bx-shef/client-bank-alfa-by  ─► инженерные issue (обезличенные, со ссылкой на отзыв)
                          │
                          ▼
        отзыв закрыт со связкой (двусторонняя трассируемость)
```

## Что делаем дальше (бэклог реализации)

Дизайн утверждён — реализацию бьём на отдельные issue (код в рамках этого документа не
пишется):

1. **Репо-приёмник** — создать приватный репо, вписать имя в `GITHUB_FEEDBACK_REPO`
   (одно место — §2 агент-дока и дефолт в `scripts/feedback-triage.sh`), завести метки
   `user-feedback` / `agent-feedback` / `feedback:{positive|problem|suggestion}`.
2. **Канал «сотрудник»** — UI 👍/👎 + комментарий на `/app`; эндпоинт приёма; автозавод
   issue с вложением файла.
3. **Канал «программа»** — из `crm-sync` заводить `agent-feedback` issue на сигналах
   `unmatched/ambiguous/manual`/не-распознан-формат; серверный дедуп + кап N/час; вложить
   файл выписки.
4. **Токены/безопасность** — `GH_WRITE_TOKEN` (fine-grained, Issues:RW, только целевые
   репо) для автозавода; read-only ingest-токен для диагностики (лимиты — агент-док §7).
5. **Косметика 👎** — продуктовое решение по нейтральной схеме оценки.
