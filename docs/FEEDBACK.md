# Обратная связь: сбор и разбор

> Last reviewed: 2026-07-22

Как в приложении импорта клиент-банка собирается обратная связь, куда она попадает и
как превращается в задачи. Этот документ — **дизайн канала** (что и куда); разбор
собранного — отдельная роль-агент, [`FEEDBACK_TRIAGE_AGENT.md`](FEEDBACK_TRIAGE_AGENT.md).

> **Статус:** **базовый канал «сотрудник» реализован** (порт из `ai-price-import`): виджет 👍/👎
> `FeedbackWidget.vue` на `/app`, роуты `GET/POST /api/feedback` (фрейм-токен), автозавод issue в
> приватный репо-приёмник (`GITHUB_FEEDBACK_*`, fail-closed без ENV) **+ телеметрия** 👍/👎 (#195:
> счётчики `feedback_up`/`feedback_down`, видны в `GET /api/import/metrics`) **+ вложение файла
> выписки по согласию** (#198: тумблер `B24Checkbox` в 👎-панели на `/import` → сырой текст выписки в
> приватный issue). Ссылка на созданную сущность (#197) **сознательно пропущена** (импорт
> fire-and-forget, `jobId→сущность` в client-bank нет) **+ программа-канал** (воркер `crm-sync`, когда
> «запуталась»: `unmatched`/`ambiguous`/`manual` → issue `agent-feedback` в приватный репо, дедуп по
> корню + кап 10/час на портал; `confusion` несёт **редактированный сэмпл** операции (ПДн, только в
> приватный репо), `fail-open`/`format` — non-PII). Разбор потока — отдельная роль-агент.

## TL;DR

- Два канала отзывов: **сотрудник** (кнопки 👍/👎 в `/app`) и **программа** (воркер
  `crm-sync`, когда «запуталась»: не нашла компанию, неоднозначное разнесение и т.п.).
- Каждый отзыв автоматически становится **issue в приватном репо** (ENV
  `GITHUB_FEEDBACK_REPO` — приватный `<org>/<private-feedback-repo>`; в код не вшит, без ENV — fail-closed). Приватность нужна, чтобы
  безопасно хранить данные клиента **и приложенный файл выписки**.
- Разбирает поток ИИ-агент по [`FEEDBACK_TRIAGE_AGENT.md`](FEEDBACK_TRIAGE_AGENT.md):
  группирует, заводит инженерные issue в `bx-shef/client-bank-alfa-by`, закрывает
  разобранное со связкой.

## Зачем отдельный приватный репо

Отзывы почти всегда несут **данные клиента**: номер счёта, суммы, УНП, названия
контрагентов, member_id портала, содержимое выписки. Такое нельзя класть в публичный
репо задач. Поэтому:

- **сырые отзывы + файлы выписок** → приватный репо-приёмник (`GITHUB_FEEDBACK_REPO`);
- **обезличенные инженерные задачи** → репо проекта `bx-shef/client-bank-alfa-by`
  (со ссылкой на приватный отзыв, где лежит конкретика).

Границу держит privacy-guard в [`FEEDBACK_TRIAGE_AGENT.md`](FEEDBACK_TRIAGE_AGENT.md)
(Шаг 5 в §3, повтор-оговорка в §5.1): агент не переносит контекст/файл из приватного
отзыва в публичный issue.

## Каналы

### 1. Сотрудник (`user-feedback`) — **реализовано**

Где: страница `/app` (под полосой статуса импорта) **и `/import`** (под разбором выписки, где
доступно вложение файла — #198). Оператор ставит оценку 👍/👎; для 👎 открывается поле комментария
«что пошло не так» (необязательное) и — если разобран файл — тумблер «приложить файл выписки»
(согласие, default OFF). 👍 уходит сразу и файла не несёт.

- UI: `app/components/FeedbackWidget.vue` (+ `useFeedback.ts`) — рендерится только когда канал
  включён на сервере (`GET /api/feedback {enabled}`), инертен вне портала.
- Транспорт: `POST /api/feedback` (фрейм-токен `Bearer`+`X-B24-Domain`, `member_id` из проверенного
  домена, `profile`-валидация) → чистый `feedbackHandler` → `buildFeedbackIssue` (санитизация:
  Trojan-Source strip + HTML-escape в `<pre><code>`, контекст в inline-code-span) → `postFeedbackIssue`
  (GitHub REST) в приватный репо `GITHUB_FEEDBACK_REPO`.
- **Durable outbox (#61) — ✅ реализовано.** Happy-path синхронный (201 → 200 с номером issue). При
  **транзиентном** сбое GitHub (5xx/429/сеть) роут кладёт уже собранный issue в очередь `feedback-post`
  (`FEEDBACK_RETRY_OPTS`: 8 попыток, backoff 30с) и отдаёт **202** — отзыв переживёт блип / закрытие
  вкладки; воркер ретраит, на успехе бампит #195-метрику, перманентный 4xx дропает. Без Redis — фолбэк на
  прежний 502. Auth+санитизация остаются **синхронно** в роуте (в очередь едет только санитизованный payload).
- Метки issue: `user-feedback` + `feedback:{up|down}`.
- 👍 (`up`) — сигнал «работает как надо»; в трекинг обычно не заводится.
- 👎 (`down`) — кандидат на инженерный issue.

> 🎛️ Косметика (из практики кита): голое 👎 демотивирует. Рассмотреть нейтральную
> схему — напр. кнопка **«🔎 Разобрать»** вместо «👎 Проблема». Решение — продуктовое,
> завести отдельным issue.

### 2. Программа — воркер `crm-sync` (`agent-feedback`) — **реализовано (MVP)**

Приложение само сигналит, **когда «запуталось»**. Сигналы считаются в `crm-sync`
(`CLAUDE.md` → счётчики `unmatched/ambiguous/manual`). После прогона воркер (best-effort,
демо-счета пропускаются) заводит **один** issue, если были «запутавшиеся» исходы:

| Сигнал | Что значит | Покрыт MVP |
|---|---|---|
| `unmatched` | компания по счёту не найдена (нет реквизита `RQ_ACC_NUM`/`RQ_IIK`) | ✅ |
| `ambiguous` | несколько кандидатов одной суммы — разнесение неоднозначно | ✅ |
| `manual` | нет точного совпадения — ушло в ручную очередь | ✅ |
| формат выписки не распознан | падает разбор `Type=X` / 1С-формат (file-parse) | ✅ |
| fail-open по стадиям | `crm.status.list` вернул пусто для известной воронки | ✅ |

- Метка issue: `agent-feedback` + `feedback:problem`. Три вида сигнала — **union `ProgramSignal`**
  (`confusion` счётчики / `fail-open` список сущностей / `format` провайдер).
- **Данные:** `fail-open` — **non-PII** (имена сущностей + `member_id` + sha). `confusion` несёт **один
  редактированный сэмпл** запутавшейся операции (направление/сумма/назначение/контрагент/счёт/УНП),
  `format` несёт **сам файл**, который не разобрался (сырой текст выписки, декод из джобы file-parse) —
  оба это **данные клиента**, уезжают **только** в приватный репо (по дизайну), рендерятся инертно
  (strip HOSTILE_CHARS/бэктиков + HTML-escape + кап `MAX_FILE_EMBED`, файл — в `<details>`). Футер issue
  честно флагует «содержит данные клиента» (триаж-агент по нему не тащит это в публичный репо). Дедуп
  держит один issue/форму/час (один сэмпл / один файл). Включая канал (`GITHUB_FEEDBACK_*`), владелец соглашается.
- **Дедуп + кап** (`server/utils/programFeedbackCap.ts`, чистое ядро над Redis, DI+тесты): дедуп по
  «корню» = **сигнатура сигнала с неймспейсом по типу** (`confusion:ambiguous+manual` /
  `failopen:deal+invoice` / `format:alfa-by` — не пересекаются), не чаще 1 раза в час на портал;
  плюс **кап 10 issue/час на портал** (Redis `INCR`-бакет), чтобы сломанный формат не завёл сотни.
- **Проводка (3 точки, все best-effort):** `confusion` — хвост `fileProgramFeedback` в `crm-sync`
  (demo-гейт по счёту); `fail-open` — в `loadNegativeStagePredicate` при `failOpenEntities.length>0`
  (fire-and-forget, только реальные порталы с токеном сюда доходят); `format` — в воркере `file-parse`
  при падении разбора (fire-and-forget + re-throw, аплоуд всегда реальный, demo-гейт не нужен).
- **Гейт:** канал включён (`GITHUB_FEEDBACK_*`, fail-closed), Redis доступен (иначе не рискуем спамить).
  Общий хелпер `fileProgramSignal` в `worker.ts`; чистые ядра — `app/utils/programFeedback.ts`
  (union-билдер issue + `programSignalSignature`) + `server/utils/programFeedbackCap.ts` (гейт).

## Формат issue-отзыва (в приватном репо)

```markdown
## Контекст
- member_id: <portal member_id>
- Канал: user-feedback | agent-feedback
- Файл выписки: <имя файла> (вложение ниже)
- Счёт/сделка: <№>, сумма: <...>, версия сборки: <sha>

## Суть
<комментарий сотрудника ИЛИ что распознала/сделала программа>

## Лог обработки
<для agent-feedback: что распознал воркер, какой счётчик сработал, стадия>
```

### Вложение файла

**Ключевое требование.** К отзыву прикладывается **сам файл выписки** (или релевантные
операции), из-за которого возникла проблема — иначе её **не воспроизвести**. Это
безопасно, потому что репо-приёмник приватный.

- Файл встраивается **в тело issue** — инертным `<pre><code>`-блоком внутри свёрнутого
  `<details>` (общий рендер `fileEmbedLines` в `app/utils/feedback.ts`): strip управляющих
  символов с сохранением переносов + `escapeHtml` (нельзя закрыть блок) + кап `MAX_FILE_EMBED`.
- Для `agent-feedback` **`format`** (файл не разобрался) файл выписки у воркера уже на руках —
  file-parse-воркер декодирует `contentBase64` из джобы в `catch` и кладёт его в `signal.fileText`.
  Для `user-feedback` — тот файл, что оператор загружал в `/import` (по согласию, тумблер `B24Checkbox`).
- Сигналы `confusion` (сэмпл операции) и `format` (файл) помечают issue футером «содержит данные
  клиента»; `fail-open` — non-PII. Privacy-guard: клиентские данные **никогда** не уезжают в
  публичный репо задач при триаже (только ссылка на приватный отзыв).

## Поток целиком

```
Сотрудник 👍/👎  ─┐
                  ├─► приватный репо-приёмник (issue + файл выписки)
Воркер crm-sync ─┘        │  метки user-feedback / agent-feedback + feedback:kind
   («запутался»)          │
                          ▼
        ИИ-агент триажа (FEEDBACK_TRIAGE_AGENT.md)
                          │  группирует по корню, сверяет с бэклогом
                          ▼
   bx-shef/client-bank-alfa-by  ─► инженерные issue (обезличенные, со ссылкой на отзыв)
                          │
                          ▼
        отзыв закрыт со связкой (двусторонняя трассируемость)
```

## Что делаем дальше (бэклог реализации)

Дизайн утверждён — реализацию бьём на отдельные issue (код в рамках этого документа не
пишется):

1. **Репо-приёмник** — приватный репо создан; задаётся через **ENV `GITHUB_FEEDBACK_REPO`**
   (формат `<org>/<private-feedback-repo>`, в код не вшит — дефолт пустой, fail-closed; реальное
   имя держим вне публичного репо). Осталось: завести метки
   `user-feedback` / `agent-feedback` / `feedback:{positive|problem|suggestion}`.
2. **Канал «сотрудник»** — ✅ **базовый готов** (UI 👍/👎 + комментарий на `/app`, эндпоинт приёма,
   автозавод issue в приватный репо) **+ телеметрия 👍/👎** (#195: `feedback_up`/`feedback_down` в
   `metrics_counter`) **+ вложение файла выписки по согласию** (#198). Ссылка на созданную сущность
   (#197) **пропущена** (импорт fire-and-forget, `jobId→сущность` нет). **Кап/дедуп
   и для сотрудника** (не только для программы) — один оператор не должен наспамить сотней issue:
   кап N/час по инициатору + дедуп по `member_id|kind` + содержимому (пока не сделан).
3. **Канал «программа»** — ✅ **готов**: `agent-feedback` issue на сигналах `unmatched/ambiguous/
   manual` (crm-sync, **+ редактированный сэмпл операции** для воспроизведения), **`format`** (падение
   разбора в file-parse, **+ приложен сам файл, который не разобрался**, инертно в `<details>`) и
   **`fail-open`** (пустая загрузка стадий, non-PII); серверный дедуп по корню (неймспейс по типу) +
   кап 10/час (Redis). Клиентские данные (сэмпл операции / файл выписки) уезжают только в приватный
   репо — футер issue честно помечает «содержит данные клиента».
4. **Токены/безопасность** — **два РАЗНЫХ пути записи, не путать:**
   - **Сбор** (каналы 1–2: сервер приложения заводит issue в **приватном** репо) — бэкенду
     нужен GitHub-креденшл с правом **Issues: RW** на приватный feedback-репо: fine-grained PAT
     (напр. `GITHUB_FEEDBACK_TOKEN` в env/секрете бэкенда) **или** GitHub App. Именно сюда идёт
     заведённый токен. Пока канал не реализован.
   - **Разбор** (триаж: ИИ-агент заводит обезличенную задачу в **публичном** репо и закрывает
     отзыв) — через **инструменты своей среды** (Claude Code / Claude-in-GitHub), отдельный PAT
     **не нужен** (MCP в проекте не настроен и не требуется).
   - Опц. read-only ingest-токен для диагностики лимитов (агент-док §7).

   **Privacy-гейт — техническим контролем, а не только инструкцией агенту:** при автозаводе
   в `PROJECT_REPO` тело issue собирается из **allowlist обезличенных полей**, файл выписки
   в публичный репо не уходит by construction (acceptance criteria п.2-3).
5. **Retention файлов выписки** (#284) — ✅ **сделано**. Приватный репо копит реальные выписки
   клиентов (ПДн), поэтому встроенный в issue **блок выписки** (`<details>…Показать содержимое…`)
   **редактируется по сроку хранения**: закрытый (разобранный) отзыв старше `FEEDBACK_RETENTION_DAYS`
   (дефолт **30 дней**, клампится [1,365]) → блок выписки заменяется на маркер
   `_[выписка удалена по retention-политике (#284)]_`, а **не-ПДн метаданные триажа остаются**
   (комментарий/счётчики/лейблы — они нужны для аналитики). Решение — чистое ядро
   `app/utils/feedbackRetention.ts` (`isPurgeable`/`redactStatementBlocks`/`planRetention`, юнит-тесты);
   исполнение — скрипт `pnpm feedback:retention` (**dry-run по умолчанию**, `--apply` пишет; листит
   закрытые issue, PATCH-ит редактированные тела) + расписание `.github/workflows/feedback-retention.yml`
   (ежедневно, `--apply`). **Идемпотентно** (повторный проход — no-op: блока уже нет) и **инертно без
   секретов** (не сконфигурирован приватный репо → exit 0). Токен PAT (Issues:RW на приватный репо) —
   секрет `FEEDBACK_RETENTION_TOKEN` воркфлоу, репо/срок — репо-переменные `GITHUB_FEEDBACK_REPO`/
   `FEEDBACK_RETENTION_DAYS`. Редактируем **тело** (снимаем ПДн), а не удаляем issue целиком — так
   разбор/аналитика по обезличенному отзыву сохраняются; при желании владелец может дополнительно
   удалять issue вручную.
6. **Косметика 👎** — продуктовое решение по нейтральной схеме оценки.
