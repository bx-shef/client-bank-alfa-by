# Общая станция телеметрии (OpenTelemetry) — #78

> Last reviewed: 2026-07-18

Единая «приёмная станция» для наблюдаемости **многих приложений** (это приложение импорта
выписок + до N других). Каждое приложение шлёт OTLP сюда, различаются по `service.name`. В
Grafana — один дашборд на все, с фильтром по сервису.

```
приложение-1 ┐
приложение-2 ├─ OTLP :4318 (bearer) ─► otel-collector ─► ClickHouse (TTL 72ч) ─► Grafana :3001
...          │
приложение-N ┘
```

Это **отдельный, самодостаточный** деплой (свой `docker-compose`, не часть SSG/CI приложения).
Папку можно вынести в собственный репозиторий как есть.

## Запуск (один раз, на мониторинг-хосте)

```bash
cp .env.example .env
# заполнить три секрета: openssl rand -hex 24 в каждый
docker compose up -d
```
- Grafana: `http://<хост>:3001` (логин `admin` / `GRAFANA_ADMIN_PASSWORD`).
- Коллектор принимает OTLP на `:4318` (HTTP) и `:4317` (gRPC), требует
  `Authorization: Bearer <OTEL_COLLECTOR_AUTH_TOKEN>`.
- ClickHouse хранит сырьё **72 часа** (`collector/config.yaml` → `ttl`).

Схему таблиц (`otel_traces`/`otel_metrics`/`otel_logs`) коллектор создаёт сам
(`create_schema: true`).

## Как завести приложение (3 строки окружения на приложение)

**Node/Nitro** (это приложение и подобные) — см. [`clients/node/README.md`](clients/node/README.md).
В приложении задать:
```
OTEL_EXPORTER_OTLP_ENDPOINT = http://<хост-станции>:4318      # БАЗА, без /v1/traces
OTEL_SERVICE_NAME           = имя-приложения
OTEL_EXPORTER_OTLP_HEADERS  = Authorization=Bearer <OTEL_COLLECTOR_AUTH_TOKEN>
```
Это приложение (импорт выписок) уже несёт бутстрап (`otel.instrument.mjs` в корне репо) — ему
нужны только эти три переменные.

**Другой стек (PHP-Bitrix и т.п.)** — подключается своим OTel-клиентом, но шлёт в **тот же**
коллектор (`:4318`, тот же bearer). OTLP — единый стандарт; PHP и Node уживаются в одной Grafana
(так устроен официальный `b24-ai-starter-otel`).

Новый сервис появляется в дашборде автоматически — в списке «Сервис» (переменная тянет
`DISTINCT ServiceName`).

## Дашборды

`grafana/dashboards/app-overview.json` — стартовый обзор: спаны/мин, ошибки/мин, p95-латентность
по операциям, топ операций по ошибкам; фильтр «Сервис» (multi). Провижинится автоматически
(папка «Telemetry»). Это **отправная точка** — правь/добавляй панели в UI (allowUiUpdates).
При первом живом прогоне проверь: (1) тип фрейма панелей (в редакторе запроса плагина выставлен
верный `Format`), (2) что временные ряды бьются по `SpanName` (мульти-серия). До первого прихода
данных панели пусты (не ошибка — `has([], …)` = нет строк).

## Приватность

Финансовых ПДн в телеметрии нет: приложения чистят на источнике (allowlist + redaction +
`portal.hash`), а коллектор **повторно** срезает чувствительные атрибуты (`transform`-процессор:
`db.statement`/URL/…). ClickHouse-ретеншн 72ч. См. `docs/PRIVACY.md` в основном репо.

## ⚠ Безопасность (прод)

- **TLS/reverse-proxy**: bearer-token — это авторизация, не шифрование. Для трафика между хостами
  поставь коллектор `:4318` и Grafana за TLS (nginx-proxy/traefik) или в приватную сеть/VPN.
- Секреты — только в git-ignored `.env` (в репозиторий не коммитим).
- Grafana/ClickHouse наружу не открывать без аутентификации/сети.

## Обслуживание

- **Секреты ClickHouse применяются только при первом старте** (пустой том `clickhouse_data`).
  Смена `CLICKHOUSE_PASSWORD` после инициализации требует ручного `ALTER USER default …` или
  пересоздания тома — иначе коллектор/Grafana не аутентифицируются.
- Диск: ClickHouse растёт с трафиком; ретеншн 72ч ограничивает. Мониторь `clickhouse_data`.
- Обновление образов: пины в `docker-compose.yml` (collector-contrib / clickhouse / grafana).
- Здоровье коллектора: `http://<хост>:13133/` (health_check).

> Живой прогон (поднять станцию, увидеть данные в Grafana) — за владельцем на сервере; в CI не
> гоняется (это отдельный деплой, не часть сборки приложения).
